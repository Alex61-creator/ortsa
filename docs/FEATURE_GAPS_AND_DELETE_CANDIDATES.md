# Недоделанные фичи и кандидаты на удаление

Последняя верификация: `2026-04-15`

Документ фиксирует технический разбор по двум категориям:

1. **Недоделанные фичи** — функциональность, которую логично довести до прод-готовности.
2. **Кандидаты на удаление** — код/файлы, которые не участвуют в runtime или потеряли актуальность.

Статус документа: рабочий аудит по текущему состоянию репозитория.

---

## 1) Недоделанные фичи (оставить и довести)

## 1.1. Управление LLM-промптами в админке

### Что уже сделано

- Backend API для CRUD промптов: `app/api/v1/admin/prompts.py`
- Frontend страница и клиент API:
  - `frontend-admin/src/pages/PromptsPage.tsx`
  - `frontend-admin/src/api/prompts.ts`

### Статус

- Подключены роутинг и меню (`/api/v1/admin/prompts`, `/prompts` и пункт меню).
- smoke `list -> edit -> reset` проведён; изменение промпта влияет на следующий запуск генерации.

### Приоритет

- **Высокий** (влияет на оперативное управление качеством LLM-отчётов).

## 1.2. Pro-эпики (roadmap-заготовка)

### Файлы

- `app/planned/pro_features.py`
- `app/product/bundle_gift_spec.py`

### Назначение

- Формализуют будущие направления: synastry/progressions/calendar, bundle/gift-модели.

### Что не доделано

- Нет runtime-интеграции (это спецификация, а не рабочая фича).
- Нет связки с задачами/эпиками в явном трекере внутри репозитория.

### Что сделать

- Либо оставить как "single source of truth" для roadmap и добавить ссылку из `docs/`.
- Либо перенести в `docs/roadmap/` (если принцип: в `app/` хранится только исполняемый код).

### Приоритет

- **Средний** (архитектурная гигиена и прозрачность roadmap).

## 2) Кандидаты на удаление

Ниже список "remove candidates" с условиями, когда удаление безопасно.

## 2.1. `app/tasks/scheduler.py`

### Почему кандидат

- Файл содержит только поясняющий docstring.
- Рабочее расписание уже сконцентрировано в `app/tasks/worker.py`.

### Условие безопасного удаления

- Команда подтверждает, что отдельный "маркерный" файл для навигации не нужен.

### Риск удаления

- Низкий (функциональный риск практически отсутствует).

---

## 2.2. `app/utils/landing_html.py`

### Почему кандидат

- В runtime-потоке приложения не используется.
- Встречается в тестах как изолированный helper.

### Условие безопасного удаления

- Подтвердить, что не планируется возврат к server-side рендеру/подстановкам для legacy HTML-лендинга.
- Удалять вместе со связанными тестами (`tests/test_landing_html.py`), чтобы CI оставался зелёным.

### Риск удаления

- Низкий-средний (риск только для сценария возврата legacy-пайплайна).

---

## 2.3. Папка `HTML макеты/` (опционально)

### Почему "условный" кандидат

- В runtime не участвует.
- Используется как дизайн-референс и артефакт согласования UX.

### Условие безопасного удаления

- Все нужные макеты перенесены в дизайн-систему/внешний источник (Figma, docs).
- Команда не использует эту папку в текущем UI-процессе.

### Рекомендация

- Чаще лучше **архивировать**, а не удалять сразу.

---

## 2.4. Legacy `standard/premium`: что считать мёртвым, а что нет

### Что найдено

- Историческая миграция с упоминанием старых кодов тарифов:
  - `alembic/versions/p1q2r3s4t5u6_merge_legacy_tariff_codes.py` (`standard/premium` -> `report/pro`)
- Legacy-статические артефакты:
  - `frontend/public/static/index.html` (`wizard-tariff--premium`, при этом `data-tariff="pro"`)
  - `frontend/public/static/css/landing.css` (CSS-классы `*-premium`)
- Дополнительно: `uvicorn[standard]` и `@standard-schema/*` — это зависимости, не тарифы.

### Классификация

- **Миграция Alembic** — не runtime-код, но важное историческое наследие схемы БД.  
  Не считать "мусором" и не удалять автоматически.
- **Legacy static с `premium`** — вероятный мёртвый код, если прод работает только через актуальный React SPA-поток.

### Условие безопасного удаления legacy static

- Подтверждено, что в проде не используются страницы/ассеты из `frontend/public/static/*`.
- Нет внешних ссылок/прокси-правил, ведущих на эти legacy-файлы.
- После удаления пройден smoke тест лендинга/чекаута на актуальном SPA.

## 3) Рекомендуемая стратегия

1. Сначала довести prompts-фичу до рабочего состояния (backend + frontend + тест).
2. После этого выполнить "чистку":
   - удалить `app/tasks/scheduler.py`;
   - удалить `app/utils/landing_html.py` + связанные тесты (если legacy не нужен).
3. Для roadmap-файлов (`pro_features`, `bundle_gift_spec`) зафиксировать политику:
   - либо оставить в `app/` как спецификации;
   - либо перенести в `docs/roadmap/`.
