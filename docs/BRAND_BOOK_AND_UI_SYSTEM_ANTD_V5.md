# ASTRO Brand Book & UI System (AI-Ready)

Версия: `v1.0.0`  
Дата: `2026-04-15`  
Статус: `Active`  
База: `Ant Design v5`  
Приоритет правил: `Brand > Ant Design defaults`

---

## 1. Назначение документа

Этот документ — единый стандарт для:

- дизайна интерфейсов;
- генерации кода UI-компонентов;
- написания тестов;
- проверки качества перед релизом.

Документ обязателен для людей и для нейросети перед созданием любого нового модуля/элемента.

---

## 2. Миссия и позиционирование бренда

### Миссия

Сделать астрологический прогноз доступным каждому: понятный и интерпретируемый результат за считанные минуты с четкими ответами на вопрос, что делать и как поступить.

### Бренд-ценности

- Точность.
- Скорость.
- Доверие.
- Престиж.

### Tone of Voice

- Прямой, ясный, человеческий язык.
- Минимум абстракций и "эзотерического тумана".
- Каждый смысловой блок должен вести к действию.

### Ядро аудитории

- Женщины 16-60.
- Мужчины 30-50.
- Работающая аудитория.

- Web.

## 4. Конкурентная рамка

## 4.1. Основные классы конкурентов

1. Классические порталы (например, `astro.com`).
2. Мобильные астроприложения.
3. Частные астрологи.

## 4.2. Наши преимущества

- Быстрее частных консультаций.
- Понятнее и современнее старых порталов.
- Практичнее контентных приложений за счет actionable-выводов.
- Сильнее в масштабируемом post-purchase опыте.

## 4.3. Что обязательно усиливать

- Объяснимость интерпретации (почему такой вывод).
- Эмпатичность языка без потери ясности.
- Доверие через прозрачную методологию и стабильное качество.

---

## 5. Дизайн-принципы

1. `Clarity first`: смысл важнее декоративности.
2. `Action-oriented`: интерфейс приводит к следующему шагу.
3. `Premium simplicity`: визуально дорого, но лаконично.
4. `Consistency`: единые паттерны во всех модулях.
5. `Accessibility by default`: читаемость и управление без барьеров.
6. `Mobile-first`: критические сценарии проектируются от мобильного опыта.

## 6.3. Нейтральная палитра (Dark)

- `bg`: `#0B1020`
- `surface`: `#121A2B`

- `border`: `#2B3852`

- Color нельзя использовать как единственный способ передачи состояния (добавлять текст/иконку).
- Контраст текста и интерактивов — минимум WCAG 2.1 AA.

### Шрифты (Google Fonts)

- Основной: `Inter`.

- `H1`: 40/48, 700
- `H2`: 32/40, 700
- `H3`: 24/32, 600
- `H4`: 20/28, 600
- `Body L`: 18/28, 400
- `Body M`: 16/24, 400
- `Body S`: 14/20, 400
- `Caption`: 12/16, 500

## 10. Component Catalog (обязательные правила)

Для каждого компонента обязательны состояния: `default`, `hover`, `focus`, `active`, `disabled`, `loading`, `error` (если применимо).

## 10.1. Button

- Типы: `Primary`, `Default`, `Text`, `Link`, `Danger`.
- Высота: 40 (M), 48 (L).
- Радиус: 12.
- Разрешено не более одного `Primary` в одном смысловом блоке.

## 10.2. Input / Select / DatePicker

- Явные лейблы обязательны.
- Placeholder не заменяет label.
- Ошибка: и цвет, и текст-пояснение.
- Формат/маски обязательны для даты и времени.

## 10.3. Form

- Поля группируются по смыслу.
- Валидация: inline + сводка ошибок в верхней части длинных форм.
- CTA формы должен отражать действие пользователя.

## 10.4. Card

- Padding: 20-24.
- Радиус: 16.
- Структура: заголовок -> интерпретация -> следующий шаг.

## 10.5. Tabs

- Не более 5 табов в 1 ряду.
- Для mobile использовать scrollable tabs.
- Названия короткие и предметные.

## 10.6. Modal / Drawer

- Для фокусной задачи; не использовать как замену полноценной странице.
- Обязательная кнопка закрытия и клавиша `Esc`.
- Длинные сценарии на mobile — через `Drawer`.

## 10.7. Table

- Использовать только для табличных сущностей.
- Прогнозы/интерпретации — через карточки, не таблицы.

## 10.8. Feedback components

- `Alert`, `Result`, `Empty`, `Skeleton`, `Spin`, `Notification`.
- Для загрузки >2 сек показывать прогресс и объяснять, что сейчас происходит.

## 11. UX-состояния модуля (mandatory)

Каждый модуль должен иметь:

1. `Loading`: что загружается и ориентир по ожиданию.
2. `Empty`: почему пусто и как получить результат.
3. `Error`: что пошло не так и что делать.
4. `Success`: подтверждение + следующий шаг.

---

## 12. Контент и микрокопирайтинг

### Формула сообщения

`Контекст -> Интерпретация -> Практический шаг -> CTA`

### Правила

- Обращение на "вы" (по умолчанию).
- Без категоричных обещаний.
- Без сложной терминологии без объяснения.
- В каждом важном блоке есть ответ: "Что сделать дальше?".

---

## 13. Доступность и качество

- Стандарт: `WCAG 2.1 AA`.
- Полная клавиатурная навигация.
- Видимый `focus ring` у интерактивных элементов.
- ARIA-метки для иконок и custom controls.
- Текст не должен ломаться на 200% zoom.

## 14. AI Protocol: как нейросеть должна работать

## 14.1. Обязательный вход перед генерацией

- Название модуля.
- Цель модуля.
- Целевая платформа.
- Основной пользовательский сценарий.
- Primary CTA.
- Список допустимых Ant-компонентов.
- Нужные состояния.
- Тема (`light`/`dark`/`both`).

## 14.2. Обязательный формат ответа нейросети

1. Краткая UX-концепция.
2. Структура экрана.
3. Список Ant-компонентов.
4. Theme tokens/overrides (если есть).
5. Код компонента.
6. Тесты (`unit`, `integration`, при необходимости `visual`).
7. Self-check соответствия бренд-буку.

## 14.3. Self-check (перед выдачей результата)

- Соответствует миссии и ценностям?
- Текст прямой, понятный и actionable?
- Есть обязательные состояния?
- Контраст/доступность соблюдены?
- Не нарушены правила компонентов и токенов?
- Решение адаптивно для mobile?

---

## 15. Definition of Done для UI-модуля

Модуль готов, если:

- соответствует этому бренд-буку;
- решает задачу пользователя минимумом шагов;
- имеет обязательные состояния;
- покрыт тестами;
- проходит доступность и не ломается в dark/light;
- не конфликтует с архитектурой Ant v5.

---

## 16. Версионирование документа

Схема: `MAJOR.MINOR.PATCH`.

- `MAJOR`: изменение бренд-принципов, миссии или базовой визуальной системы.
- `MINOR`: новые паттерны, компоненты, расширения правил.
- `PATCH`: уточнение формулировок, мелкие исправления.
