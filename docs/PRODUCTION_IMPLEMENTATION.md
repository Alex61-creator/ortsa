### Соответствие текущему коду (сводка)

Ниже — соотнесение рисков с репозиторием (справочно). **Порядок работ** — по **приоритетному чеклисту** выше; детали — в §1.8–§2.3.

| Тема | В коде сейчас | Зазор / план |
|------|----------------|--------------|
| Секреты, `SECRET_KEY`, Redis | `SECRET_KEY` обязателен в настройках; в Compose Redis с `requirepass` | Прод: **никогда** не деплоить с дефолтами из `.env.example`; отдельные секреты per env; хранить в CI/GitLab/Vault |
| ЮKassa: двойная обработка | Дедуп по ключу в Redis (`notification_idempotency_key` + TTL 24 ч); `Order.yookassa_id` — **UNIQUE** в БД | Персистентный аудит: таблица `webhook_log` (payload); при сбросе Redis дубликат теоретически возможен до истечения бизнес-идемпотентности |
| Ссылки из писем | Письма: `…/reports/{order_id}`; редирект со старого `/cabinet/orders/:id` во фронте | На сервере: финальный смоук после деплоя |
| Caddy / TLS | `Caddyfile` с заглушкой домена | Реальный домен, DNS до ACME, проверка HTTP→HTTPS |
| Alembic | `alembic upgrade head` в entrypoint контейнера | Стратегия **expand/contract** для миграций без окна несовместимости (§1.9) |
| Celery | `task_acks_late=True`, `result_expires=3600`, soft/hard time limits | В dev `docker-compose.yml` нет `stop_grace_period`; в **`docker-compose.prod.yml`** — 90 с; health очереди не в `/health` |
| DeepSeek / LLM | `AsyncOpenAI` + `timeout` (`LLM_HTTP_TIMEOUT_SECONDS`) + tenacity | Алерт по очереди: поля в `/health/ready` при `HEALTH_CHECK_CELERY`, либо Flower |
| Health | `/health` — liveness; `/health/ready` — БД + Redis; при `HEALTH_CHECK_CELERY` — ping воркеров и длина очереди | В проде задайте `HEALTH_CHECK_CELERY=true` |
| Rate limiting | slowapi: лимиты на auth и создание заказов (`RATE_LIMIT_*`) | При росте — уточнить лимиты и ключ (IP vs user id) |
| Пул PostgreSQL | `DB_POOL_SIZE` / `DB_MAX_OVERFLOW` в настройках | Подстроить в `.env` под сумму воркеров + `max_connections` PG |
| JWT | `jwt.decode(..., algorithms=[settings.JWT_ALGORITHM])` | Поддерживать явный список алгоритмов при любых изменениях |
| Mass assignment | OAuth создаёт `User` с явными полями; `UserCreate` — email/password | При появлении регистрации по API — не пробрасывать `**dict` в модель |
| Файлы в storage | Пути формируются из `order_id` / служебных префиксов | Любые пути из внешних данных — нормализовать и запретить `..` |
| ПДн в логах | structlog | OAuth редирект с `?token=` — риск попадания токена в access-логах прокси (§2.3) |
| SPF/DKIM/DMARC | Упомянуто в фазе деплоя | Операционный чеклист DNS + mail-tester (Unisender: кабинет + DNS домена) |

## 1. Бэкенд и эксплуатация

### 1.1. Docker Compose: dev vs prod

**Суть:** в репозитории Compose удобен для разработки (`./app`, `./static` смонтированы в контейнер). В продакене обычно **не** монтируют исходники: работает **собранный образ**, тома — только для данных (БД, `storage`, при необходимости конфигов).

| Статус | Действие |
|--------|----------|
| [x] | Описать **prod-override** или отдельный `docker-compose.prod.yml`: без volume на `./app` и `./static` для сервиса API (или только read-only артефакты, если принято). См. корневой `docker-compose.prod.yml`. |
| [x] | Сборка образа в CI/CD: `docker build` с тегом по коммиту/релизу (GitHub Actions: job `docker-build`). |
| [ ] | Секреты в проде: **Docker secrets**, **Vault**, переменные оркестратора — не копировать `.env` на сервер вручную без политики. |

---

### 1.2. Каталог `storage` и права (Dockerfile `chmod 777`)

**Суть:** в `Dockerfile` для каталога storage заданы широкие права — для продакена предпочтительны **непривилегированный пользователь** в образе и **том** с корректными UID/GID на хосте.

| Статус | Действие |
|--------|----------|
| [ ] | Ввести пользователя `USER app` (или аналог) в Dockerfile после установки зависимостей; каталог `storage` создавать с владельцем этого пользователя. |
| [ ] | Убрать `chmod 777` или заменить на `chown` нужному пользователю + `chmod 750`/`770` по политике. |
| [ ] | В Compose для прода: именованный volume или bind-mount с **uid/gid**, совпадающими с пользователем в контейнере. |

---

### 1.3. Uvicorn: один процесс и масштабирование

**Суть:** по умолчанию поднимается **один** процесс Uvicorn; под ростом нагрузки нужны **несколько воркеров** или несколько реплик за балансировщиком.

| Статус | Действие |
|--------|----------|
| [ ] | Зафиксировать целевой RPS/латентность и решение: `uvicorn --workers N` (с учётом CPU и async), либо **несколько контейнеров** + load balancer. |
| [ ] | Убедиться, что **sticky sessions** не требуются для API (JWT stateless — обычно не нужны). |
| [ ] | При нескольких воркерах: **Celery** уже выносит тяжёлую работу — оценить узкие места (БД, Redis, LLM). |

---

### 1.4. Секреты и безопасность конфигурации

**Суть:** в `.env.example` — заглушки. **Нельзя** выкатывать прод с теми же значениями, что в примере: `SECRET_KEY` подписывает JWT; при слабом или общем со стейджингом ключе возможна подделка токена и вход от имени любого пользователя, включая админа.

**Принципы:** отдельные секреты для **dev / staging / prod** — без переиспользования; хранение в **переменных CI/CD** (GitHub Secrets, GitLab Variables), **Vault** или секретах оркестратора — не копировать «как есть» чужой `.env` на сервер.

| Статус | Действие |
|--------|----------|
| [ ] | Сгенерировать криптостойкий `SECRET_KEY` для прода (и отдельный для staging). Черновик: `scripts/generate_production_secrets.sh`. |
| [ ] | Уникальные пароли БД и Redis в проде; Redis с **`requirepass`** (в `docker-compose.yml` уже задаётся через `REDIS_PASSWORD`). |
| [ ] | Хранение секретов: менеджер секретов / переменные окружения платформы; ротация по регламенту. |
| [x] | Проверить `.gitignore`: `.env`, ключи, `*.pem` не коммитятся. Дополнительно: `scripts/check_tracked_secrets.sh` и ручной просмотр `git ls-files`. Если секрет уже попал в историю — **`git filter-repo`**. |
| [ ] | В проде не полагаться на «скопировал `.env.example` в `.env` и запустил» — значения задаются только из безопасного канала. |

---

### 1.5. Админка (SQLAdmin)

**Суть:** доступ по `ADMIN_USERNAME` / `ADMIN_PASSWORD` из env — для прода нужна **политика паролей** и **ограничение доступа** (сеть, VPN, отдельный поддомен, IP allowlist на reverse proxy).

| Статус | Действие |
|--------|----------|
| [ ] | Длинный случайный пароль админа; отдельная учётная запись не для личных целей. |
| [ ] | Путь `/admin` доступен только с **доверенных IP** или через **VPN** / **SSO** (если появится). |
| [ ] | HTTPS только; при возможности — отдельный хост `admin.example.com` с жёстким firewall. |

### 1.5.1. Админ-SPA (отдельный поддомен)

**Суть:** панель на React (`frontend-admin/`), OAuth Google для админки (`/api/v1/auth/google/authorize-admin`), JWT с проверкой `is_admin`. Allowlist в env: `ADMIN_GOOGLE_EMAILS`, `ADMIN_TELEGRAM_USER_IDS` — при входе выставляется `is_admin` в БД. API под префиксом `/api/v1/admin/*`. SQLAdmin (`/admin`) остаётся резервным. Полный визуальный и навигационный охват как в **`astrogen-admin-v2.html`** не обязателен для первого прода — см. раздел «Текущая стадия» выше.

| Статус | Действие |
|--------|----------|
| [ ] | Задать `ADMIN_APP_ORIGIN` (origin админ-SPA) и добавить его в CORS (приложение добавляет origin автоматически, если задан). |
| [ ] | В reverse proxy: хост `admin.example.com` → статика сборки `frontend-admin` + прокси `/api/*` на FastAPI (как основной сайт). |
| [ ] | Ограничить доступ к поддомену админки (VPN / allowlist), не полагаться только на OAuth. |
| [ ] | Заполнить allowlist админов в env; при смене состава админов — обновить env и при необходимости снять флаг `is_admin` в БД вручную. |
| [ ] | Собрать `frontend-admin` (`npm run build`), выкласть `dist/` на хост админки; локальный запуск и чеклист проверки — в корневом [`README.md`](../README.md) (раздел «Админ-панель»). |

### 1.6. Резервное копирование и retention

**Суть:** приложение не пишет бэкапы само. В репозитории есть **скрипты** для типичного деплоя на Docker Compose (том `postgres_data` и `astro_storage`). **Расписание** (cron), **retention** файлов в каталоге бэкапов и **копия вне сервера** (S3, второй диск) — задача эксплуатации.

| Статус | Действие |
|--------|----------|
| [x] | **PostgreSQL:** [`scripts/backup_postgres.sh`](../scripts/backup_postgres.sh) — `pg_dump -Fc` в `backups/astro_pg_<timestamp>.dump.gz` (контейнер `postgres` должен быть запущен). Managed Postgres у облачного провайдера — используйте встроенные снапшоты вместо или вместе с этим. |
| [x] | **Файлы storage:** [`scripts/backup_storage.sh`](../scripts/backup_storage.sh) — `tar.gz` содержимого `/app/storage` в `backups/astro_storage_<timestamp>.tar.gz` (через `docker compose run --no-deps fastapi`). |
| [x] | **Проверка дампа БД:** [`scripts/verify_pg_backup.sh`](../scripts/verify_pg_backup.sh) — `pg_restore -l` без записи в БД. |
| [ ] | **Расписание:** на сервере задать cron (пример: [`deploy/cron.backup.example`](../deploy/cron.backup.example)); частота — по целевому RPO (часто раз в сутки для малого трафика). |

#### 1.6.1. Учебный прогон восстановления (staging / тестовая БД)

Цель: убедиться, что архивы открываются, БД поднимается из дампа, файлы storage согласованы с ожиданиями — **без** риска для единственной копии прода.

1. **PostgreSQL:** на отдельном хосте или в новом проекте Compose с **пустой** БД той же major-версии Postgres. Проверка списка объектов: `./scripts/verify_pg_backup.sh путь/к/дампу.dump.gz`. Полное восстановление в тестовую БД `restore_drill` (подставьте `POSTGRES_USER` из `.env` в окружение оболочки, например `export $(grep -E '^POSTGRES_USER=' .env | xargs)`):

   ```bash
   docker compose -f docker-compose.prod.yml --env-file .env exec -T postgres \
     createdb -U "$POSTGRES_USER" restore_drill
   gunzip -c backups/astro_pg_YYYYMMDD_HHMMSS.dump.gz | \
     docker compose -f docker-compose.prod.yml --env-file .env exec -iT postgres \
     pg_restore -U "$POSTGRES_USER" -d restore_drill --no-owner --no-acl
   ```

   После проверки: `docker compose … exec -T postgres dropdb -U "$POSTGRES_USER" restore_drill`. На проде **не** выполнять поверх боевой БД без согласованного окна обслуживания и свежего снапшота.

2. **Storage:** распаковать архив в временный каталог и сравнить число файлов / выборочно открыть PDF; путь в контейнере при работе приложения — `STORAGE_DIR` (в Compose прод: `/app/storage`).

3. **Зафиксировать** дату прогона и участников (чеклист для аудита).

### 1.10. Celery: graceful shutdown, LLM, очередь

**Суть:** при `docker stop` контейнер воркера получает SIGTERM; без достаточного grace period длинная задача (генерация PDF + LLM) обрывается — заказ может остаться в `processing`, пользователь без отчёта.

**В коде сейчас (`app/tasks/worker.py`):** `task_acks_late=True`, `worker_prefetch_multiplier=1`, soft/hard time limits (25/30 мин), `result_expires=3600`. В **`docker-compose.yml`** (dev) нет **`stop_grace_period`** у воркера; в **`docker-compose.prod.yml`** задано **90 с**.

**DeepSeek:** задан `LLM_HTTP_TIMEOUT_SECONDS` для `AsyncOpenAI`; tenacity по-прежнему на rate limit / API errors.

| Статус | Действие |
|--------|----------|
| [x] | Задать **`stop_grace_period`** (например 60–120 с) для `celery-worker` в Compose/K8s, согласовать с `task_soft_time_limit`. См. `docker-compose.prod.yml` (90 сек для worker). |
| [x] | Явный **HTTP timeout** для вызовов DeepSeek (`LLM_HTTP_TIMEOUT_SECONDS`); circuit breaker — опционально позже. |
| [x] | Длина очереди в **`/health/ready`** (`celery_queue_length`) при `HEALTH_CHECK_CELERY`; опционально **503** при превышении (`CELERY_QUEUE_FAIL_READINESS`). Flower/Prometheus — дополнительно. |
| [x] | **`/health/ready`:** при `HEALTH_CHECK_CELERY` — ping воркеров (`celery_workers_ok`). |

### 2.2. Caddy и TLS

**Суть:** в `Caddyfile` для продакена указан заглушечный домен **`yourdomain.com`**. Если не заменить — ACME (Let's Encrypt) не выпустит сертификат на чужой домен: Caddy может **не поднять TLS** или **не стартовать** в ожидаемой конфигурации.

## 5. Пошаговый план вывода в прод

Ниже — **практическая последовательность** от нуля до приёма реальных пользователей. Сроки и детали (managed DB vs Postgres на том же VPS) зависят от бюджета и нагрузки.

### Фаза A: домен, сервер, сеть

1. **Купить домен** у регистратора (или использовать существующий). Включить DNSSEC по желанию.
2. **Выбрать хостинг:** VPS (выделенный виртуальный сервер) или PaaS. Минимум для старта — 2 vCPU, 4 GB RAM (с запасом под Postgres + Redis + API + Celery + Caddy; при росте — вынести БД в managed-сервис).
3. **Создать DNS-записи:** `A` / `AAAA` на IP VPS для основного сайта (`www` и `@` по политике); отдельно `A` для `admin.` если админка на поддомене.
4. **SSH:** только ключи, отключить вход по паролю для root; по возможности не использовать root для деплоя (отдельный пользователь `deploy` с `docker` group).

### Фаза B: безопасность и базовое ПО

5. **Firewall:** открыть 22 (SSH с ограничения по IP), 80, 443; закрыть всё остальное. Postgres/Redis **не** торчать наружу, если они на том же хосте — только `127.0.0.1` или docker network.
6. **Установить Docker Engine и Docker Compose plugin** (или оркестратор по выбору).
7. **Fail2ban** (опционально) для sshd.

### Фаза C: приложение и секреты

8. **Репозиторий:** клонировать на сервер или собирать образы в CI и тянуть только образы (предпочтительно для прода).
9. **Секреты прод:** сгенерировать `SECRET_KEY`, пароли БД/Redis, заполнить ЮKassa, OAuth (redirect URI на **прод-домен**), `DEEPSEEK_API_KEY`, SMTP, `TELEGRAM_BOT_TOKEN`, allowlist админов. Хранить вне git (env-файл с правами 600, или секреты CI → deploy).
10. **`PUBLIC_APP_URL` и `SITE_URL`:** канонический HTTPS-URL без завершающего слэша, совпадающий с тем, что видит пользователь.
11. **Миграции:** `alembic upgrade head` перед первым запуском трафика.
12. **Тарифы/сид:** при необходимости `scripts/seed_tariffs.py` или актуализация через админку/SQLAdmin.

### Фаза D: reverse proxy и TLS

13. **Caddy / Nginx:** TLS через Let's Encrypt (в `Caddyfile` заменить заглушечный домен), проксирование на контейнер API, раздача статики фронта и лендинга.
14. **Админ-SPA:** сборка `frontend-admin`, выкладка `dist/`, `ADMIN_APP_ORIGIN` и CORS.
15. **Проверить** HTTP→HTTPS редирект и HSTS по политике безопасности.

### Фаза E: внешние интеграции

16. **ЮKassa:** зарегистрировать прод-return URL и URL вебхука; проверить подпись/верификацию IP (как в конфиге).
17. **OAuth провайдеры:** Google/Yandex/Apple — callback URLs на прод-домен.
18. **Почта:** SPF/DKIM/DMARC для домена отправки писем.

### Фаза F: наблюдаемость и данные

19. **Бэкапы:** cron + `BACKUP_ROOT` вне git; скрипты §1.6; копирование `storage` и pg_dump (или managed backup у провайдера БД).
20. **Мониторинг:** uptime на `/health`, опционально Sentry; алерты по диску и 5xx.
21. **Смоук-тест:** регистрация/логин → заказ → оплата (тестовый платёж в ЮKassa) → генерация отчёта → письмо → ссылка из письма.

### Фаза G: запуск

22. **Канареечный трафик** (опционально): ограничить видимость по IP или фичефлагом.
23. **Go-live:** снять ограничения, мониторить первые часы/дни.
24. **Пост-релиз:** закрыть оставшиеся пункты **приоритетного чеклиста** и дорожную карту админки ([`ADMIN_PANEL_FUTURE.md`](./ADMIN_PANEL_FUTURE.md), паритет с `astrogen-admin-v2.html` по приоритету продукта).
