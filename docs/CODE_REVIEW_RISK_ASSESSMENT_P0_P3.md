# Code Review: ошибки и потенциальные уязвимости (P0-P3)

Дата: 2026-04-22 (дополнение red team и блоки «закрытие уязвимости» — в тот же день)  
Репозиторий: `Astro`  
Область: backend (`FastAPI`), frontend (`React`), admin frontend, LLM/PDF pipeline, infra (`Caddy`, `docker-compose`)

## Шкала приоритета

- `P0` - критично: высокий риск компрометации данных/аккаунтов или полной недоступности, требует немедленного исправления.
- `P1` - высокий: существенная уязвимость или частые ошибки бизнес-процесса, исправлять в ближайший спринт.
- `P2` - средний: риск ограничен контекстом, но технический долг уже влияет на безопасность/надёжность.
- `P3` - низкий: hardening и улучшения практик, плановое исправление.

## Findings

### P1-1: JWT передается в URL query (`token=...`) после OAuth callback
- Риск: утечка токена через логи, browser history, рефереры, сторонние аналитики и reverse-proxy access logs.
- Где: `app/api/v1/auth.py` (редирект `.../auth/callback?token=...`).
- Сценарий:
  - токен попадает в историю браузера и в URL-логи прокси/мониторинга;
  - при наличии сторонних скриптов или утечки логов возможен захват сессии.
- Рейтинг: `P1` (высокая вероятность утечки при реальной эксплуатации).
- **Закрытие уязвимости (решение):**
  - перейти на flow без токена в URL (например, одноразовый code + exchange endpoint);
  - как минимум использовать URL fragment (`#token=`) вместо query;
  - сократить TTL access token и добавить refresh-token с ротацией;
  - проверить маскирование query-string в логах proxy/app.

### P1-2: JWT хранится в `localStorage` в user/admin SPA
- Риск: при XSS токен читается JavaScript-кодом и уходит злоумышленнику.
- Где:
  - `frontend/src/stores/authStore.ts`
  - `frontend-admin/src/stores/authStore.ts`
  - `frontend/src/main.tsx` (миграция в persisted storage).
- Сценарий: любая XSS в SPA или подключенном скрипте = полный захват пользователя/админа до истечения токена.
- Рейтинг: `P1`.
- **Закрытие уязвимости (решение):**
  - перейти на `HttpOnly + Secure + SameSite` cookies для сессии;
  - если быстрый переход невозможен, минимизировать TTL и ввести принудительную ротацию токена;
  - усилить CSP и убрать inline-скрипты (см. отдельный finding).

### P1-3: SQLAdmin auth на статичном логин/пароль из env без rate-limit/2FA
- Риск: brute-force/credential stuffing на административном контуре.
- Где: `app/admin/auth.py`.
- Сценарий: при утечке/подборе пароля злоумышленник получает админ-доступ через SQLAdmin.
- Рейтинг: `P1`.
- **Закрытие уязвимости (решение):**
  - отключить SQLAdmin login в production или ограничить IP allowlist;
  - добавить rate-limit + временные блокировки + аудит неуспешных попыток;
  - перейти на единый OAuth/JWT admin flow с RBAC.

### P2-1: `StorageService.resolve_path()` разрешает абсолютные пути
- Риск: потенциальный path traversal/чтение произвольных файлов при компрометации DB-полей путей или ошибках валидации.
- Где:
  - `app/services/storage.py` (`resolve_path` возвращает абсолютный путь как есть);
  - использование в `app/api/v1/reports.py` при `FileResponse`.
- Сценарий: если `report.pdf_path`/`chart_path` окажется абсолютным внешним путем, API может отдать файл вне storage.
- Рейтинг: `P2` (эксплуатация требует дополнительного шага, но impact высокий).
- **Закрытие уязвимости (решение):**
  - запретить absolute paths полностью;
  - canonicalize path и проверять `is_relative_to(STORAGE_DIR)`;
  - хранить только storage keys, а не filesystem absolute paths.

### P2-2: доверие `X-Forwarded-For` без явной trusted proxy-валидации в коде
- Риск: обход IP allowlist webhook-проверки при ошибочной сетевой конфигурации.
- Где:
  - `app/utils/client_ip.py`
  - `app/api/v1/webhooks.py` (проверка YooKassa IP).
- Сценарий: если endpoint окажется доступен напрямую, клиент может подделать `X-Forwarded-For`.
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  - извлекать IP только от доверенных proxy-сетей;
  - на периметре блокировать прямой доступ к backend;
  - добавить defense-in-depth в коде (trusted proxies list).

### P2-3: CSP допускает `'unsafe-inline'` для script/style
- Риск: повышает эксплуатабельность XSS.
- Где: `app/middleware/security_headers.py`.
- Сценарий: при наличии XSS-поинта CSP с `'unsafe-inline'` не блокирует выполнение инъекции.
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  - перейти на nonce/hash-based CSP;
  - убрать inline JS/CSS из лендинга и auth страниц;
  - поэтапно ввести `Content-Security-Policy-Report-Only` и отчеты.

### P2-4: prompt injection — пользовательский текст в промпте к LLM
- Риск: манипуляция поведением модели (обход структуры отчёта, нежелательный/токсичный контент в PDF, вставка посторонних секций). Это не RCE и не SQLi, но влияет на целостность продукта и доверие; при цепочке с PDF (см. P2-5) усиливается impact.
- Где:
  - `app/services/llm.py` — `build_user_prompt`: при `LLM_USE_KERYKEION_CONTEXT` в user попадает XML из `to_context` (данные субъекта из Kerykeion);
  - `app/services/synastry_llm.py` — `build_synastry_user_prompt`: имена двух людей явно вставляются в текст user-сообщения перед XML/JSON;
  - `app/utils/sanitize.py` — `sanitize_string` убирает HTML/опасное для Jinja, но **не** ограничивает естественно-языковые инструкции к модели.
- Сценарий: пользователь вводит в `full_name` / имя для синастрии / иные поля, попадающие в контекст, текст вида «игнорируй предыдущие инструкции…».
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  - явно разделять инструкции и данные (один помеченный блок «ниже только данные карты, не инструкции»; по возможности структурировать данные JSON без свободного префикса);
  - пост-валидация ответа (обязательные маркеры секций, отказ перегенерации при нарушении);
  - политики модели/фильтры на стороне провайдера, где доступны;
  - для синастрии: не интерполировать имена как свободный текст, а передавать как экранированные поля структуры (например JSON key `person_1_label`).

### P2-5: ответ LLM в PDF как «сырой» HTML (`md_to_html` + `| safe`, Jinja `autoescape=False`)
- Риск: модель может вернуть HTML-теги; `_inline_markup` в `app/services/pdf.py` обрабатывает только `**`/`*`, не экранирует `<...>`. В `app/templates/pdf/report.html` результат идёт через `| md_to_html | safe`. Окружение PDF в `PDFGenerator` создаётся с `autoescape=False`. Итог: произвольная разметка внутри генерируемого PDF (вплоть до попыток внешних ресурсов — зависит от WeasyPrint и конкретных конструкций).
- Где: `app/services/pdf.py`, `app/templates/pdf/report.html`.
- Сценарий: prompt injection или сбой модели → вредоносный/фишинговый фрагмент внутри файла отчёта.
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  - после Markdown-преобразования пропускать HTML через белый список тегов или экранировать всё, кроме разрешённых `<p>`, `<strong>`, `<em>`, `<ul>`, `<li>`;
  - либо включить autoescape и явно помечать только безопасные фрагменты;
  - убрать `| safe` для необработанного вывода модели.

### P2-6: system prompt для LLM из БД, редактируемый через admin API
- Риск: при компрометации админской учётки (JWT admin, см. P1-2) или инсайдере — полная подмена system-инструкций для отчётов по тарифу/локали (в т.ч. эксфильтрация стиля данных, вредоносные инструкции модели для всех пользователей тарифа).
- Где: `app/services/prompt_templates.py`, `app/api/v1/admin/prompts.py`, использование в `app/tasks/report_generation.py`.
- Рейтинг: `P2` (зависит от компрометации admin-контура; по impact сопоставимо с целостностью генерации).
- **Закрытие уязвимости (решение):** вести аудит изменений промптов в БД; ограничить право правки RBAC; для production — опционально dual-control (два согласующих); лимиты длины полей; алерты в мониторинг при изменении критичных шаблонов.

### P3-1: возможная конкурентная проблема в OAuth state consume
- Риск: race-condition при очень редком конкурентном повторе callback.
- Где: `app/services/oauth_state.py`.
- Комментарий: сейчас `get` + `delete` не атомарно, лучше `GETDEL`/Lua script.
- Рейтинг: `P3`.
- **Закрытие уязвимости (решение):** сделать одноразовое потребление state атомарным.

### P3-2: использование `MD5` в `cache.make_key`
- Риск: в данном контексте не криптографическая функция безопасности, но лучше избегать MD5 в новых реализациях.
- Где: `app/core/cache.py`.
- Рейтинг: `P3`.
- **Закрытие уязвимости (решение):** заменить на `SHA-256` для единообразия security-practice.

## Дополнение: red team / повторный обзор (после условного закрытия пунктов выше)

Ниже — уязвимости и риски, выявленные при повторном анализе «как злоумышленник» (контроль сервиса, финансы, доступность, целостность данных). У каждого пункта указано **конкретное закрытие**.

### P1-4: дублирующий эндпоинт возврата без проверки `can_refund` (обход RBAC)
- Риск: любой пользователь с `is_admin=true`, но без права на возврат (или с компрометированным «слабым» админским JWT), инициирует полный/частичный возврат в ЮKassa и архивацию отчёта; прямой финансовый ущерб и отзыв задач Celery.
- Где: `POST /api/v1/orders/{order_id}/refund` в `app/api/v1/orders.py` (`Depends(get_current_admin_user)`); канонический путь с RBAC — `POST /api/v1/admin/orders/{order_id}/refund` в `app/api/v1/admin/orders.py` (`get_current_admin_user_can_refund` + `append_admin_log`).
- Сценарий: скомпрометированная учётка с флагом админа без `can_refund` вызывает «старый» маршрут под префиксом `/orders/.../refund` и обходит ограничение.
- Рейтинг: `P1`.
- **Закрытие уязвимости (решение):**
  1. Удалить `refund_order` из `app/api/v1/orders.py` **или** заменить зависимость на `get_current_admin_user_can_refund` и добавить такой же `append_admin_log`, как в админском маршруте (предпочтительно **один** публичный контракт — только `/admin/orders/.../refund`).
  2. Обновить фронт админки и документацию API, чтобы нигде не вызывался удалённый URL.
  3. (Опционально) временно на периметре (WAF/Caddy) запретить `POST /api/v1/orders/*/refund` до выката патча.

### P1-5: rate limit SlowAPI по `request.client` за reverse proxy (общий «ведро» / DoS)
- Риск: за Caddy/Docker все запросы к Uvicorn часто видны с **одного** IP прокси → лимит `RATE_LIMIT_PER_MINUTE` делят все пользователи; атакующий исчерпывает квоту и **блокирует API для всех**; либо лимит per-IP на реального клиента не работает.
- Где: `app/core/rate_limit.py` (`key_func=get_remote_address`); прокси: `docker-compose.prod.yml` + `Caddyfile`.
- Сценарий: массовые запросы к `/api/v1/auth/...`, `/geocode`, и т.д. с одного NAT до бэкенда → 429/отказ для легитимного трафика.
- Рейтинг: `P1` (доступность и косвенная защита от brute-force).
- **Закрытие уязвимости (решение):**
  1. Реализовать свой `key_func` для Limiter: брать IP из **первого** доверенного `X-Forwarded-For` (или `Forwarded`) только если `request.client.host` входит в список доверенных прокси (тот же подход, что для `client_ip` / вебхуков), иначе — только `request.client.host`.
  2. Убедиться, что Caddy передаёт `X-Forwarded-For` (по умолчанию передаёт); при необходимости задать `trusted_proxies` в конфиге приложения и единый источник правды для IP.
  3. Задать **разные** лимиты для `/auth/*`, `/webhooks/*`, `/geocode`, остального API (чтобы узкое место не «убивало» весь сервис).

### P2-7: path traversal через **относительные** сегменты (`..`) в путях storage
- Риск: при записи в БД вредоносного `pdf_path`/`chart_path` (SQLAdmin, инсайдер, будущий баг) `Path(STORAGE_DIR) / path` или `base_dir / path` без нормализации даёт выход из каталога хранилища → чтение/отдача файлов вне storage.
- Где: `app/services/storage.py` (`resolve_path`); `app/api/v1/reports.py`; `app/api/v1/admin/reports.py` (`Path(settings.STORAGE_DIR) / ...`); любые места с конкатенацией без `resolve()` + проверки границы.
- Сценарий: значение вида `reports/../../../etc/passwd` после разрешения указывает вне `/app/storage`.
- Рейтинг: `P2` (нужен шаг записи в БД; impact высокий).
- **Закрытие уязвимости (решение):**
  1. Ввести единую функцию `safe_storage_path(raw) -> Path`: запретить абсолютные `raw`; взять `(STORAGE_DIR / raw).resolve()` и проверить `resolved.is_relative_to(STORAGE_DIR.resolve())`, иначе 404/500 без утечки деталей.
  2. Использовать её во **всех** `FileResponse`, вложениях email, SQLAdmin-скачиваниях.
  3. В SQLAdmin ограничить редактирование `pdf_path`/`chart_path` или убрать поля из формы, оставив только read-only ключи из пайплайна генерации.

### P2-8: allowlist админов из env только повышает права, не понижает
- Риск: после удаления email/id из `ADMIN_GOOGLE_EMAILS` / `ADMIN_TELEGRAM_USER_IDS` пользователь с `is_admin=true` в БД **остаётся** админом навсегда; уволенный/скомпрометированный админ сохраняет полный доступ через JWT/SQLAdmin (если доступен).
- Где: `app/services/admin_allowlist.py` (`if user.is_admin: return` до проверки allowlist).
- Сценарий: ротация персонала или реакция на инцидент только через env не срабатывает без ручного SQL.
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  1. Добавить ветку: если пользователь админ **только** из allowlist (флаг в БД или сравнение с env), а сейчас **не** в allowlist — выставить `is_admin=false` (и при необходимости инкремент `token_version` для инвалидации JWT).
  2. Для «вечных» суперадминов не из env — отдельное явное поле/роль в БД с аудитом, не смешивать с env-only логикой.
  3. Документировать операционную процедуру: при offboarding — revoke в БД + смена секретов при компрометации.

### P2-9: вебхук `payment.succeeded` для заказа не сверяет сумму платежа с `order.amount`
- Риск: при аномалии на стороне провайдера, регрессе в создании платежа или ручном вмешательстве заказ может быть помечен `PAID` при несовпадении фактически оплаченной суммы и суммы заказа (нарушение бухгалтерской целостности).
- Где: `app/services/payment.py` → `process_webhook_event` (ветка `order_id`, `payment.succeeded`).
- Сценарий: редкий edge-case интеграции; защита «глубины обороны».
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  1. После загрузки заказа по `order_id` сравнить `Decimal(payment_obj["amount"]["value"])` с `order.amount` (и валюту с `RUB`); при несовпедении — лог `critical`, не переводить в `PAID`, алерт в мониторинг.
  2. Для частичных сценариев (если появятся) — явная отдельная модель статусов и согласование с ЮKassa.
  3. Покрыть интеграционным тестом: webhook с неверной суммой не меняет статус.

### P2-10: публичный `/geocode` как открытый relay к Nominatim
- Риск: злоупотребление квотой/ToS OSM, блокировка IP приложения у Nominatim, деградация продукта; при п.1-5 — усиление DoS по одному ключу лимита.
- Где: `app/api/v1/geocode.py` (без авторизации).
- Сценарий: боты гоняют запросы через ваш API → внешний сервис режет вас.
- Рейтинг: `P2`.
- **Закрытие уязвимости (решение):**
  1. Требовать авторизацию для `/geocode` **или** отдельный низкий лимит + капча/turnstile на лендинге перед вызовом.
  2. Кэшировать ответы по нормализованному `q` (Redis) с TTL, дедупликация одинаковых запросов.
  3. Согласовать с политикой Nominatim (rate, идентификация); при росте нагрузки — свой геокодер или платный API с ключом.

### P2-11: `chmod 777` на `/app/storage` в образе
- Риск: любой процесс под любым UID в контейнере может писать/удалять артефакты; при компрометации одного процесса — порча или подмена PDF/PNG.
- Где: `Dockerfile` (`chmod 777 /app/storage`).
- Рейтинг: `P2` (контейнерный hardening).
- **Закрытие уязвимости (решение):**
  1. Запускать Uvicorn/Celery от одного непривилегированного пользователя; `chown -R appuser:appgroup /app/storage` и права `0750`/`0640` вместо `777`.
  2. Убедиться, что volume в проде наследует корректного владельца (init-container или `user:` в compose).
  3. Проверить, что запись в storage не требует world-writable (если требует — исправить пайплайн, а не расширять права).

### P2-12: один `SECRET_KEY` на подпись JWT и сессию SQLAdmin
- Риск: единая точка компрометации: утечка/слабый ключ одновременно ломает и токены API, и cookie-сессию SQLAdmin (`AdminAuth(secret_key=...)`).
- Где: `app/core/config.py`, `app/core/security.py`, `app/admin/__init__.py` / `AdminAuth`.
- Рейтинг: `P2` (defense in depth / криптопрактика).
- **Закрытие уязвимости (решение):**
  1. Ввести отдельный `SQLADMIN_SESSION_SECRET` (или `ADMIN_UI_SECRET_KEY`) в env, использовать **только** его в `AdminAuth(..., secret_key=...)`.
  2. Оставить `SECRET_KEY` для JWT и прочих подписей приложения; ротация по отдельным процедурам.
  3. В production отключить SQLAdmin (см. P1-3) или вынести за VPN/IP allowlist — тогда blast radius ниже.

### P3-3: публичный `openapi.json` — полная карта API для разведки
- Риск: упрощает поиск скрытых/дублирующих маршрутов (в т.ч. исторических), параметров и сценариев атаки без чтения исходников.
- Где: `app/main.py` (`openapi_url=.../openapi.json`).
- Рейтинг: `P3`.
- **Закрытие уязвимости (решение):**
  1. В `production` отключить OpenAPI: `openapi_url=None` при `ENVIRONMENT == "production"` (или отдельный флаг `ENABLE_OPENAPI=false`).
  2. Либо отдавать схему только с внутренних IP / basic-auth для команды.
  3. Периодически ревьюить дубли маршрутов и «legacy» эндпоинты по результатам схемы в dev/staging.

## Что проверено дополнительно

- Прямых признаков SQL injection в reviewed-коде не выявлено (используется SQLAlchemy ORM/Core).
- Проверка webhook содержит полезный слой сверки через YooKassa API (`verify_*_matches_api`) - это плюс к защите.
- Основные роуты user data/report download используют проверку владельца заказа.
- CORS не выставлен как `*` в приложении, что корректно.
- HTML-письма (`EmailService._render_template`): Jinja с `autoescape=True` — вывод LLM в `monthly_digest.html` в `<pre>` не даёт «классического» HTML-XSS в теле письма так же, как PDF-шаблон с `| safe`.

## Итог по риску проекта

- `P0`: не выявлены в просмотренной области (в т.ч. после дополнения red team).
- `P1`: **5** проблем — исходные три (P1-1 … P1-3) плюс **обход RBAC на refund** (P1-4) и **rate limit за reverse proxy** (P1-5); блок P1-4 … P1-5 требует первоочередного исправления с точки зрения денег и доступности API.
- `P2`: **12** проблем — исходные шесть (P2-1 … P2-6) плюс дополнение (P2-7 … P2-12): относительный path traversal, снятие админки с allowlist, сверка суммы вебхука с заказом, злоупотребление geocode, права на storage, разделение секретов JWT/SQLAdmin.
- `P3`: **3** проблемы — исходные две (P3-1, P3-2) плюс открытый OpenAPI в prod (P3-3).

Общий профиль риска: **средне-высокий** из-за цепочки `token in URL + localStorage + admin basic auth`, **контура LLM→PDF**, а после дополнения — из-за **дублирующего refund-маршрута** и **некорректного ключа rate limit** за прокси (финансы и доступность).

## Рекомендуемый порядок remediation

**Срочно (до/вместе с релизом, если дополнение ещё не закрыто в коде):**

1. Устранить дубль `POST /api/v1/orders/.../refund` или выровнять RBAC и аудит (P1-4).
2. Исправить `key_func` SlowAPI и разнести лимиты по классам маршрутов (P1-5).

**Далее — исходный план и усиление:**

3. Убрать JWT из URL callback и перевести веб-сессию на cookie-based auth (P1-1).
4. Закрыть/ограничить SQLAdmin auth в production (P1-3); отдельный секрет для сессии SQLAdmin (P2-12).
5. Жестко ограничить file path: абсолютные запретить, относительные нормализовать и `is_relative_to(STORAGE_DIR)` (P2-1 + P2-7).
6. Ввести trusted proxy логику для client IP (P2-2); согласовать с п.2 (P1-5).
7. Сверка суммы в вебхуке `payment.succeeded` с `order.amount` (P2-9).
8. Ужесточить CSP и убрать `unsafe-inline` (P2-3).
9. Синхронизация allowlist админов с понижением прав при исключении из env (P2-8).
10. Geocode: авторизация/кэш/отдельные лимиты (P2-10).
11. Dockerfile: убрать `chmod 777`, непривилегированный пользователь и владелец volume (P2-11).
12. Закрыть P3: atomic state consume, hash SHA-256, отключить или ограничить `openapi.json` в prod (P3-1 … P3-3).
13. Смягчить prompt injection (P2-4); безопасная цепочка LLM→PDF (P2-5); аудит промптов из админки (P2-6).
