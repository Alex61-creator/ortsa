# Замечания по ревью бэкенда и план доработок

Документ зафиксирован для последующей реализации. Статусы задач можно вести в issue-трекере или чеклистом внизу.

---

## 1. Создание заказа и сбой ЮKassa (`create_order`)

**Проблема:** Заказ коммитится в БД до вызова `Payment.create`. Если ЮKassa вернёт ошибку, остаётся заказ в `PENDING` без рабочей оплаты (или в неконсистентном состоянии).

**План внедрения:**

1. Выбрать стратегию (одну или комбинацию):
   - **A.** Транзакция: сначала создать платёж во внешнем API, затем в одной транзакции записать заказ + `yookassa_id` (если API позволяет порядок — обычно наоборот: локальный заказ нужен для `metadata.order_id`; тогда компенсация).
   - **B.** Компенсация: при ошибке `create_payment` — удалить заказ или перевести в отдельный статус `FAILED_TO_INIT_PAYMENT` и дать пользователю «повторить оплату».
   - **C.** Фоновая задача: периодически чистить/напоминать по «висящим» `PENDING` без `yookassa_id` старше N минут.
2. Реализовать выбранный вариант в `app/api/v1/orders.py` + при необходимости миграция/enum статуса.
3. Добавить тест: мок падения `Payment.create` → ожидаемое поведение (откат или статус).
4. Обновить OpenAPI/описание для фронта (если появятся новые статусы или эндпоинт повтора оплаты).

## 2. Гонки по вебхуку `payment.succeeded`

**Проблема:** Два почти одновременных уведомления могут оба пройти проверку `order.status == PENDING` и вызвать `generate_report_task.delay` дважды.

**План внедрения:**

1. В `process_webhook_event` (или обёртке) загружать заказ с **`SELECT … FOR UPDATE`** (в SQLAlchemy: `with_for_update()`), в одной транзакции с коммитом перевода в `PAID` и постановкой задачи.
2. Альтернатива/дополнение: уникальный ключ или флаг «уже поставлена генерация» на уровне БД.
3. Тест: два параллельных вызова обработчика (или имитация) → одна задача Celery / идемпотентность бизнес-логики.

## 3. Ссылка на скачивание в письме «отчёт готов»

**Проблема:** `download_link` строится как `{YOOKASSA_RETURN_URL}/reports/{order.id}` — это URL фронта, не REST API. При несовпадении маршрутов SPA пользователь получит 404.

**План внедрения:**

1. Ввести в конфиг отдельную переменную, например **`PUBLIC_APP_URL`** или **`FRONTEND_BASE_URL`** (без привязки к «return URL» оплаты).
2. В `app/tasks/report_generation.py` и шаблоне письма использовать эту базу + согласованный с фронтом путь (например `/cabinet/orders/{id}` или `/reports/{id}`).
3. Задокументировать в `.env.example` и README для фронтенда.
4. Проверить письмо в тестовом окружении ЮKassa/почты.

## 4. Удаление аккаунта и JWT (`DELETE /users/me`)

**Проблема:** После анонимизации токен остаётся действительным до истечения `exp`.

**План внедрения:**

1. Зафиксировать продуктовое решение: **достаточно для MVP** или нужна **инвалидация**.
2. Если нужна инвалидация: версия пользователя в JWT + проверка в `get_current_user`, или blocklist в Redis по `jti`/user_id, или короткий TTL токенов.
3. Миграция БД при необходимости (`token_version` на `users`).
4. Тесты: после `DELETE /me` старый токен → 401.

## 5. Дублирование engine в Celery (`report_generation`)

**Проблема:** Отдельный `create_async_engine` в воркере — дублирование пула и настроек.

**План внедрения:**

1. Вынести создание engine/session в общий модуль (например `app/db/session.py` с фабрикой или `app/db/celery_session.py`), чтобы воркер использовал ту же конфигурацию URL и пул по политике (размер пула для worker отдельно при необходимости).
2. Убедиться, что при shutdown воркера соединения закрываются корректно.
3. Регрессионный прогон генерации отчёта в staging.

---

## 6. Apple Sign in — режим callback

**Проблема:** Реализован GET callback; у Apple может быть `response_mode=form_post`, тогда нужен POST.

**План внедрения:**

1. Сверить настройки App ID / Services ID в Apple Developer с текущими redirect URI и response mode.
2. При необходимости добавить `POST /apple/callback`, разбор `application/x-www-form-urlencoded`.
3. Ручной или автоматизированный тест входа через Apple в staging.

---

## 7. Логирование HTTP-исключений

**Проблема:** Все `HTTPException` попадают в лог как `error` — шум от 401/404.

**План внедрения:**

1. В `app/core/exceptions.py` в обработчике `http_exception_handler` логировать **`warning`** для 4xx клиента и **`error`** только для 5xx (или не логировать 404 для ожидаемых путей).
2. Согласовать список «тихих» кодов с командой.
3. Проверить объём логов на staging под нагрузкой.

---

## 8. Нормализация `metadata.order_id` в вебхуке

**Проблема:** Значение из API может быть строкой или числом; редкий крайний случай — нестандартный формат → ошибка при `int(order_id)`.

**План внедрения:**

1. В `process_webhook_event` безопасно приводить `order_id`: `try/except ValueError`, логирование и выход без 500.
2. Юнит-тест с разными типами `metadata`.
