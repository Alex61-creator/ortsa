# Аналитика в админке: цели, блоки, метрики и статус внедрения

Версия: `v1.0`  
Дата: `2026-04-16`  
Статус: `Living document` — обновлять при появлении новых экранов/API и событий.

---

## 1. Зачем этот документ

Свести в одном месте:

- какую **ключевую аналитику** нужно иметь для оценки эффективности бизнеса, рекламы, промокодов и продукта;
- как она **логически делится** на три блока (глобально / разовые отчёты / подписки);
- что **уже собирается** в системе (события, заказы, настройки) и что **уже есть в UI/API админки**;
- куда **дорасти**: метрики, графики, выгрузки CSV.

**Интерактивные чекбоксы** — в разделе [§5](#5-чеклисты-внедрения-gfm-интерактивные) (синтаксис GitHub: `- [x]` сделано, `- [ ]` не сделано). Для позиций «наполовину» используется неотмеченный пункт с пометкой **(частично: …)** в тексте.

В таблицах §2–4 колонка «Статус» дана словами (**Да** / **Частично** / **Нет**) — так читается и в PDF/печати; живые галочки смотрите в §5.

Источники истины в коде (для поддержки документа):

- События: `app/models/analytics_event.py`, запись — `app/services/analytics.py`, схема имён — `docs/ANALYTICS_EVENT_SCHEMA.md`.
- Growth / воронка / когорты: `app/services/event_based_metrics.py`, `app/api/v1/admin/metrics.py`, упрощённая воронка — `app/api/v1/admin/funnel.py`.
- Дашборд: `app/api/v1/admin/dashboard.py`, `app/services/order_ops_metrics.py`.
- Серверный CSV: `app/api/v1/admin/export.py`.
- Заказы/платежи/пользователи: `app/api/v1/admin/orders.py`, `payments.py`, `users.py`.
- Промокоды (CRUD, не глубокая аналитика): `app/api/v1/admin/promos.py`, UI — `frontend-admin/src/pages/PromosPage.tsx`.
- Тумблеры доп. разделов отчёта: `orders.report_option_flags`, цены — `app_settings` (см. `app/services/report_option_pricing.py`); UI — `/report-options`.

Экспорт CSV: кнопки «↓ CSV» и «BOM» в шапке (`frontend-admin/src/layouts/AdminLayout.tsx`) — первая `<table>` на странице (`frontend-admin/src/utils/exportTableCsv.ts`); серверные выгрузки — `GET /api/v1/admin/export/*.csv`.

## 2. Блок A — Общая (глобальная) аналитика по проекту

Цель: здоровье воронки, экономика, каналы, когорты, инфраструктура, сводные деньги без жёсткой привязки к типу тарифа.

### 2.1 Метрики и срезы

| Нужно для бизнеса | Метрика / срез | Источник данных | Статус |
|-------------------|----------------|-----------------|--------|
| Воронка signup → оплата → отчёт | CR1, шаги воронки, конверсии между шагами | `analytics_events` + `compute_funnel_steps` / `compute_growth_metrics` | Да: Growth, metrics `/overview` и event-funnel; Частично: страница `/funnel` (упрощённые шаги) |
| Экономика | AOV, attach-rate add-on, LTV/CAC, contribution margin, CAC по каналам | события + `orders` + `marketing_spend_manual` | Да: `/growth`, `/metrics/economics`, `/metrics/overview` |
| Когорты удержания | M1 / M3 / M6 по когортам | `compute_retention_cohorts` | Да: `/growth` |
| Маркетинговые расходы | Ручной spend по каналам/периодам | `marketing_spend_manual` | Да: `/growth` |
| Операционные риски по заказам | failed, processing > 2h и т.д. | `order_ops_metrics` | Да: дашборд |
| Свод пользователей/заказов/денег | users_total, суммы paid/refund, мини-история по месяцам | `users`, `orders` | Да: `business_metrics` / `llm_metrics` на дашборде из реальных полей заказов; **mrr_history** и AI по месяцам — из `orders` |
| Сегментация рекламы | UTM + `source_channel` в событиях | `AnalyticsEvent` | Да: витрина `/campaigns` и блок трафика на дашборде (последние 30 дней) |
| Инфраструктура | DB, Redis, Celery, … | health checks | Да: `/health` |
| Аудит изменений | Кто что менял в админке | `admin_action_log` + API | Да: `/log` |

### 2.2 Графики (желательно)

| График | Статус |
|--------|--------|
| Мини-воронка на дашборде | Да |
| MRR / выручка по месяцам (столбцы на дашборде) | Да (суммы заказов по месяцу; не классический MRR подписок) |
| Трафик-источники (декоративный блок) | Да: топ каналов по `signup_completed` (дашборд) |
| Когортная heatmap | Да: `/growth` — heatmap-ячейки M1/M3/M6 |
| Воронка Growth (event-based) | Да: API; Да: на `/growth` — ступенчатая визуализация шагов |

### 2.3 CSV и выгрузки

| Возможность | Статус |
|-------------|--------|
| Экспорт первой таблицы на странице в CSV | Да: кнопка в шапке |
| Серверная выгрузка произвольного среза (период, канал, тариф) | Да: `/admin/export/*.csv` (заказы, подписки, кампании; фильтр `billing_type` у заказов) |
| Единый формат UTF-8 BOM для Excel | Да: `excel_bom=1` на сервере и кнопка BOM в шапке админки |

## 3. Блок B — Единоразовые отчёты (one-time)

Сюда относятся разовые покупки: `report`, `bundle`, add-on отчёты, `synastry_addon`, прочие `billing_type = one_time` и т.д. Цель — конверсия в разовый чек, AOV, допродажи (add-on, тумблеры), качество промо **на чек**.

### 3.1 Метрики

| Нужно | Метрика | Источник | Статус |
|-------|---------|----------|--------|
| Объём разовых продаж | Выручка, число заказов, AOV по one-time | `orders` ⋈ `tariffs` | Да: `/one-time-sales` и API `/admin/metrics/one-time-monthly` |
| Базовые отчёты | Заказы `report` / `bundle`, статусы, суммы | `orders`, `tariffs` | Да: список заказов; Да: `report_option_flags` и оценка строки в API списка |
| Тумблеры доп. разделов | Attachment, 1–4 опции, популярность по ключу, оценка выручки строки тумблеров | `report_option_flags`, `app_settings` | Да: `/report-options` и API `/admin/metrics/report-options-analytics` |
| Add-on синастрия | attach-rate в Growth | события `addon_attached` + eligible из заказов | Да: Growth |
| Промокоды | Список, лимиты, on/off | `promocodes` | Да: `/promos` |
| Эффективность промокода | Redemptions, скидка vs выручка, конверсия по коду | `promocode_redemptions`, `orders.promo_code` | Да: `/promo-analytics` |
| Качество рекламы для разовых | `source_channel` / UTM | события, пользователь | Да: `/campaigns` с `billing_segment=one_time` и `/admin/metrics/campaign-performance/one-time` |

### 3.2 Графики

| График | Статус |
|--------|--------|
| Динамика разовой выручки по месяцам (только one_time) | Да: `/one-time-sales` |
| Популярность тумблеров (bar по ключам) | Да: `/report-options` |
| Распределение заказов 0/1/2/3/4 тумблера | Да: `/report-options` |

### 3.3 CSV

| Возможность | Статус |
|-------------|--------|
| Таблица заказов / платежей → CSV через шапку | Да (если первая таблица на странице — основная) |
| Выгрузка заказов с полями promo, тариф, сумма, флаги тумблеров | Да: API заказов + `/admin/export/orders.csv` |

## 4. Блок C — Подписки (Astro Pro)

Цель: новые подписки, продления (если отражены в данных), отвал, выручка по подписочным тарифам, связь с LLM/стоимостью.

### 4.1 Метрики

| Нужно | Метрика | Источник | Статус |
|-------|---------|----------|--------|
| Первые оплаты Pro | Заказы с `tariffs.billing_type = subscription` | `orders` | Да: `/subscriptions` |
| Новые подписки | Созданные `subscriptions` | `subscriptions` | Да: `/subscriptions` и API overview |
| Продления без заказа | Webhook продления | `payment.py`, `analytics_events` | Да: событие `subscription_renewal_payment` |
| Удержание | Retention M1 в Growth | события + когорты | Да: `/growth` (общая когортная логика) |
| Синастрия для Pro | квота / безлимит | кабинет + API синастрии | Частично: ссылка на пользователей с `/subscriptions`; единого Pro-dashboard нет |

### 4.2 Графики

| График | Статус |
|--------|--------|
| Подписочная выручка по месяцам (только subscription) | Да: `/subscriptions` (заказы + продления) |
| Active subscriptions over time | Частично: текущий «active now» + месячные ряды; без исторического снимка active по месяцам |

### 4.3 CSV

| Возможность | Статус |
|-------------|--------|
| Как у блока A/B — первая таблица | Да |
| Выгрузка подписок с периодами | Да: `/admin/export/subscriptions.csv` и список в `/subscriptions` |

## 6. Рекомендуемые следующие шаги (без обязательства к срокам)

1. **Сегментированный отчёт по месяцам**: one_time vs subscription + блок report/bundle upsell из `orders` (и пометка про продления без заказа).
2. **Расширить `AdminOrderListItem`**: `promo_code`, `report_option_flags` (read-only) для саппорта и CSV с заказов.
3. **Промо-дашборд**: агрегаты по `promocode_redemptions` + связка с заказами.
4. **Серверный `GET .../export.csv`** для ключевых таблиц с параметрами `from`, `to`, `segment`.
