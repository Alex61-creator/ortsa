# Бизнес-логика AstroGen

Документ описывает поведение системы по коду бэкенда (FastAPI) и фронтенда (React SPA). Актуальность: соответствует реализации в репозитории.

---

## 1. Общая модель продукта

### 1.1. Сущности

- **Пользователь** — регистрация через OAuth (Google, Yandex, Apple) или Telegram Web App; выдаётся JWT. Для расчётов и заказов требуется согласие с политикой (`consent_given_at`).
- **Натальные данные** — сохранённый профиль: ФИО, дата/время рождения, место, координаты, таймзона, опционально система домов, язык отчёта (`report_locale`: `ru` / `en`). Лимит профилей на пользователя зависит от тарифных возможностей.
- **Тариф** — код, цена, привязка к уровню ИИ-интерпретации (`llm_tier`), срок хранения отчёта, для части тарифов — подписка (`billing_type`).
- **Заказ** — связка пользователь + натальные данные + тариф; статус жизненного цикла; для платных — интеграция с ЮKassa; опционально `report_delivery_email` для доставки PDF и чека.
- **Отчёт** — PDF (и путь к PNG карты), статус; создаётся после успешной оплаты или сразу для бесплатного тарифа.

### 1.2. Жизненный цикл заказа

Статусы заказа логически следуют цепочке: `PENDING` (создан, ожидает оплаты) → при успешной оплате `PAID` → `PROCESSING` (идёт генерация) → `COMPLETED` или `FAILED`. Возможны отмена оплаты (`CANCELED`), сбой инициализации платежа и др.

**Бесплатный тариф** (`цена <= 0`, код `free`): заказ сразу получает статус оплаченного, в очередь Celery ставится задача генерации отчёта без прохода через ЮKassa. Повторное использование бесплатного тарифа одним пользователем запрещено политикой `free_order_policy`.

**Платный тариф**: создаётся платёж в ЮKassa, клиент перенаправляется на `confirmation_url`. После успешной оплаты вебхук ЮKassa переводит заказ в `PAID` и ставит ту же задачу генерации в Celery.

### 1.3. Оплата и вебхуки

- Создание платежа: `YookassaPaymentService.create_payment` с метаданными (`order_id`, `tariff`).
- HTTP-уведомления: `POST /webhooks/yookassa` — проверка IP (опционально), идемпотентность через кэш, при `payment.succeeded` обновление заказа и постановка `generate_report_task`. Для тарифов-подписок при необходимости создаётся/обновляется подписка и сохраняется способ оплаты.

### 1.4. Доставка отчёта и email-политика

- Адрес для чека и отчёта выводится из email аккаунта и/или явного `report_delivery_email` (см. `resolve_receipt_and_report_email`): для аккаунтов без «реальной» почты обязателен явный email для отчёта и чека.

### 1.5. Прочие API-области (кратко)

- Каталог тарифов, профиль пользователя, экспорт данных, геокодирование для места рождения, подписки (статус / отмена / возобновление), служебные метрики для админов — отдельные модули. **Ежемесячный дайджест** (если включён) обрабатывается фоновыми задачами Celery, отдельного публичного REST для него нет.
- Ядро разового PDF — цепочка заказ → расчёт → LLM → PDF → почта; скачивание готового файла — см. раздел 4 и п. 2.10 ниже.

## 2. SPA-воронка: от данных до отправки отчёта

Ниже — **логика работы React SPA**: отправка данных в FastAPI, затем (асинхронно) расчёт карты, вызов **DeepSeek API** для текста, сборка PDF и отправка письма.

### 2.1. Вход в воронку

| Канал | Описание |
|--------|----------|
| **React SPA** | Страницы `/order/tariff` → `/order/data` → `/order/confirm`: те же эндпоинты через `createNatalData` и `createOrder`. |

### 2.2. Шаг 1 — Авторизация (обязательна до отправки данных)

- **OAuth**: редирект на `/api/v1/auth/.../authorize`.
- **Telegram Web App**: `POST /api/v1/auth/twa` с `initData` → в ответе `access_token` сохраняется в `sessionStorage`.

Без JWT запросы к защищённым эндпоинтам (натальные данные, заказы) не выполняются.

### 2.3. Шаг 2 — Геокод и сбор полей

- Клиент запрашивает `GET /api/v1/geocode/?q=...` и получает координаты, таймзону и т.д. (`__geocodeResult` в клиентском UI-потоке).
- Вводятся имя, дата, время, место; для продвинутых тарифов — система домов (в SPA).

### 2.4. Шаг 3 — Отправка данных в FastAPI: натальные данные

**Запрос:** `POST /api/v1/natal-data/` с телом JSON:

- `full_name`, `birth_date`, `birth_time`, `birth_place`, `lat`, `lon`, `timezone`
- `accept_privacy_policy: true` (фиксирует согласие пользователя)
- `report_locale`: `ru` или `en` (с языка интерфейса / `localStorage`)

**Бизнес-правила на сервере:**

- Проверка согласия и лимита профилей.
- Санитизация строк (имя, место).
- Запись строки `NatalData` в БД.

**Ответ:** объект с `id` созданной записи — он нужен для заказа.

### 2.5. Шаг 4 — Создание заказа в FastAPI

**Запрос:** `POST /api/v1/orders/` с телом:

- `tariff_code` — выбранный тариф
- `natal_data_id` — id из предыдущего шага
- `report_delivery_email` — для платных тарифов при необходимости (альтернативный email для доставки и чека)

**Логика:**

- Проверка согласия пользователя, существования тарифа и принадлежности натальных данных пользователю.
- Расчёт email для чека/отчёта; при невозможности — ошибка 400 с текстом про необходимость указать почту.
- **Бесплатный тариф:** заказ `PAID` → сразу `_queue_free_report(order.id)` → Celery `generate_report_task.delay(order_id)`.
- **Платный:** заказ `PENDING` → создание платежа ЮKassa → `confirmation_url` в ответе; клиент открывает ссылку (в TWA через `openLink`). Генерация отчёта **не** стартует до подтверждения оплаты.

### 2.6. Шаг 5 — Триггер генерации (платные)

После успешной оплаты ЮKassa шлёт вебхук; обработчик помечает заказ `PAID` и вызывает `generate_report_task.delay(order_id)` — та же задача, что и для бесплатного тарифа.

### 2.7. Фоновая задача: расчёт карты (не DeepSeek)

Сервис `AstrologyService` (Kerykeion): по дате, времени, координатам и таймзоне строится натальная карта, SVG → PNG, в кэш по ключу параметров кладётся структура карты для повторного использования. PNG сохраняется в объектное хранилище через `StorageService`.

**Важно:** DeepSeek не считает астрономию — в LLM уходят **уже рассчитанные** данные карты (JSON).

### 2.8. Вызов DeepSeek API (LLM)

Класс `LLMService`:

- Клиент OpenAI-совместимый: `base_url=https://api.deepseek.com`, ключ `DEEPSEEK_API_KEY`, модель из настроек (`LLM_MODEL`).
- По тарифу определяется **уровень** (`LlmTier`: free / pro / full): разный system prompt (объём секций), разные лимиты `max_tokens`, для PRO в user prompt добавляется запрос учесть транзиты.
- User prompt: сериализованные данные натальной карты (`json.dumps(chart_data)`).
- Ответ проходит разбор в структурированный вид (`LLMResponseSchema.from_markdown`), результат кэшируется на длительный срок.

Итог — текст интерпретации по разделам (маркеры `## [...]`), готовый для вёрстки PDF.

### 2.9. Формирование PDF

`PDFGenerator` подставляет в шаблон (`report_free.html` для free-уровня, иначе `report.html`):

- локаль, ФИО, дата/место рождения, путь к PNG карты, текст интерпретации, название тарифа, время генерации.

Файл сохраняется, запись `Report` обновляется (`ACTIVE`), заказ — `COMPLETED`.

### 2.10. Отправка отчёта и скачивание

`EmailService` отправляет письмо на:

- `order.report_delivery_email`, если задан, иначе email пользователя.

Шаблон письма зависит от локали (`report_ready.html` / `report_ready_en.html`), тема содержит номер заказа, **вложение — PDF**. В шаблон подставляется `download_link`: URL вида `{public_app_base_url}/reports/{order_id}` (канонический маршрут React; старый путь `/cabinet/orders/:id` редиректит на `/reports/:id`).

**Прямая выдача файла по API (Bearer JWT):**

- `GET /api/v1/reports/{order_id}/download` — PDF (заказ пользователя, статус завершён, отчёт активен).
- `GET /api/v1/reports/{order_id}/chart` — PNG карты при тех же условиях.

При ошибке на любом этапе генерации заказ может перейти в `FAILED`, у отчёта — статус ошибки; задача Celery имеет повторы при исключениях.

## 3. Сводная схема цепочки (SPA → отчёт)

```
[Клиент: авторизация]
        ↓
POST /natal-data/     → сохранение профиля рождения
        ↓
POST /orders/         → заказ (бесплатный: сразу очередь | платный: ЮKassa)
        ↓
[Платный: webhook payment.succeeded] → очередь Celery
        ↓
generate_report_task
        ↓
AstrologyService      → расчёт карты + PNG
        ↓
LLMService (DeepSeek) → текст интерпретации
        ↓
PDFGenerator          → PDF на диск / storage
        ↓
EmailService          → письмо с PDF и ссылкой в кабинет
```

## 4. Полный перечень эндпоинтов FastAPI (`/api/v1`)

Префикс задаётся `API_V1_STR` (по умолчанию `/api/v1`). Ниже — маршруты, согласованные с `app/api/v1/__init__.py` и роутерами.

| Префикс | Методы и пути | Назначение |
|--------|----------------|------------|
| `/auth` | `POST /twa` | Telegram Web App: `initData` в JSON → JWT |
| | `GET /google/authorize`, `GET /google/callback` | OAuth Google |
| | `GET /yandex/authorize`, `GET /yandex/callback` | OAuth Yandex |
| | `GET /apple/authorize`, `GET /apple/callback`, `POST /apple/callback` | OAuth Apple (POST — ответ form_post от Apple) |
| `/natal-data` | `POST /`, `GET /`, `GET /{id}`, `PATCH /{id}`, `DELETE /{id}` | CRUD натальных профилей |
| `/orders` | `GET /`, `GET /{order_id}`, `POST /` | Список, карточка, создание заказа |
| | `POST /{order_id}/refund` | Возврат (только админ API) |
| `/tariffs` | `GET /` | Публичный каталог тарифов |
| `/reports` | `GET /{order_id}/download`, `GET /{order_id}/chart` | PDF и PNG по готовому заказу |
| `/users` | `GET /me`, `PATCH /me`, `GET /me/export`, `DELETE /me` | Профиль, согласие, экспорт, удаление аккаунта |
| `/webhooks` | `POST /yookassa` | Уведомления ЮKassa |
| `/geocode` | `GET /?q=...` | Геокодинг (обязательный query `q`, min 2 символа) |
| `/subscriptions` | `GET /me`, `POST /me/cancel`, `POST /me/resume` | Подписка Astro Pro |
| `/ops` | `GET /metrics/orders` | Метрики заказов (только админ API) |
| `/system` | `GET /health`, `GET /ready` | Liveness и readiness под префиксом API (дубль `/health` и `/health/ready` у корня) |

Вне префикса `/api/v1`: `GET /health`, `GET /health/ready` (готовность БД и Redis) и редиректы legacy-путей на SPA — см. `app/main.py`.
