# План поэтапного развития платформы Astro

Версия: `v0.1`  
Статус: `Working plan`

---

## 1) Контекст

Текущая реализация продукта фактически соответствует первому модулю `astro.com/natal` (натальная карта).

Следующий этап развития:
- запустить второй самостоятельный модуль `astro.com/numer` (нумерология);
- параллельно развивать основной портал `astro.com` (контент, блог, витрина тарифов);
- объединить продукты на уровне общей авторизации, биллинга и тарификации;
- перейти к зонтичной архитектуре для масштабирования новых модулей.

Ключевой принцип: **продукты остаются независимыми по доменной логике, но используют единые платформенные сервисы (auth, billing, entitlements, profile).**

---

## 2) Целевая модель

Платформа разделяется на два слоя:

1. **Product layer (вертикали)**  
   `natal`, `numer`, далее новые продуктовые модули.

2. **Platform layer (общие сервисы)**  
   - Identity/SSO;
   - Billing/Subscriptions;
   - Tariffs/Entitlements (доступ к модулям и лимитам);
   - User profile/account;
   - Общая аналитика и аудит.

Это позволяет выпускать продукты независимо, не размножая реализацию авторизации и оплаты.

---

## 3) Фазы внедрения

## Фаза A — Два автономных продукта + подготовка платформы

Цель: стабильно иметь 2 работающих модуля (`natal`, `numer`) без жесткой сцепки.

Задачи:
- довести и стабилизировать `astro.com/natal`;
- реализовать `astro.com/numer` как самостоятельный продуктовый контур;
- зафиксировать общий `Product Contract`:
  - единый формат доступа к пользователю (`/v1/users/me`);
  - единый формат получения прав доступа (entitlements);
  - единые события аналитики по продуктам;
  - единый подход к обработке платного/бесплатного доступа.

Definition of Done:
- `natal` и `numer` работают независимо;
- контракты интеграции документированы и используются в обоих модулях;
- добавление 3-го модуля не требует переписывать auth/billing в существующих.

---

## Фаза B — Единый platform core (auth + billing + тарификация)

Цель: единый аккаунт и единая монетизация для обоих модулей.

Задачи:
- внедрить SSO-сессию на доменном уровне (`Domain=.astro.com`, `HttpOnly`, `Secure`, `SameSite=Lax`);
- унифицировать flow логина/логаута для всех фронтов;
- внедрить единую модель тарифов и **entitlements** (что доступно пользователю, а не только что куплено);
- объединить подписки, платежную историю и управление тарифом в единый billing-контур;
- реализовать миграцию от локальной JWT-модели к общей сессии без downtime.

Definition of Done:
- пользователь логинится один раз и работает в `natal` и `numer` без повторного входа;
- права доступа и лимиты централизованы;
- апгрейд/даунгрейд тарифа моментально отражается в обоих продуктах.

---

## Фаза C — Основной портал + контентный контур

Цель: превратить `astro.com` в главный entrypoint экосистемы.

Задачи:
- развивать `astro.com` как портал: главная, продуктовая витрина, pricing, CTA в модули;
- запустить блог в подпапке `astro.com/blog` как канонический SEO-контур;
- синхронизировать маркетинговые и продуктовые сценарии (единый user state);
- ввести контентные и кампанийные маршруты без ломки продуктовых URL.

Definition of Done:
- `astro.com` генерирует органический трафик и конвертирует в продукты;
- блог индексируется как часть основного домена;
- пользовательский путь от контента к покупке проходит в единой сессии.

---

## Фаза D — Зонтичная архитектура и масштабирование

Цель: стандартизированное подключение новых продуктов.

Задачи:
- внедрить шаблон подключения нового модуля (роутинг, auth, billing, analytics, observability);
- выделить `static.astro.com` под файловую/CDN-зону;
- при необходимости добавить `landing.astro.com` для кампаний с риск-изоляцией;
- внедрить обязательные NFR-политики (security headers, HSTS, trace-id, readiness, audit);
- оформить ADR по спорным решениям (signed URL vs backend proxy, политика admin perimeter, canonical login entrypoint).

Definition of Done:
- новый модуль подключается по стандартному чеклисту;
- инфраструктура не требует ad-hoc решений под каждый запуск;
- доменная структура соответствует архитектурному документу.

---

## 4) Приоритеты реализации

1. `numer` как второй автономный модуль.  
2. Единый platform core (SSO + entitlements + billing).  
3. Портал и блог на `astro.com`.  
4. Инфраструктурная полировка зонтичной архитектуры (`static`, `landing`, стандарты подключения модулей).

---

## 5) Риски и как снижать

- **Риск:** начать с UI-портала раньше стабилизации platform core.  
  **Снижение:** сначала auth/billing/entitlements, потом полная UX-склейка.

- **Риск:** дублирование логики оплаты в продуктах.  
  **Снижение:** только централизованный billing API и единый каталог тарифов.

- **Риск:** сложная миграция авторизации.  
  **Снижение:** staged rollout через feature flags и совместимость старой/новой сессии на переходный период.

- **Риск:** SEO-каннибализация между контентом и продуктовыми страницами.  
  **Снижение:** каноникал, единая URL-политика и блог строго в `astro.com/blog`.

## 7) Шаблон подключения нового модуля (Module Onboarding Template)

Этот шаблон фиксирует, как подключать любой новый продуктовый модуль (`<new-product>`) к platform core без ad-hoc решений.

Базовый принцип:
- модуль независим в доменной логике и UI;
- auth/billing/entitlements/analytics/observability переиспользуются из платформы;
- все интеграции проходят через единые контракты, уже используемые в `natal` и целевом `numer`.

Архитектурный принцип для всех новых модулей:
- админка, БД, billing, доставка отчетов и storage/S3 остаются едиными платформенными контурами;
- внутри модуля меняются только доменные контракты (`<new-product>` endpoints), типы запросов к LLM и pipeline-логика;
- разные модули могут иметь полностью разные pipeline (например, `natal` и `taro`) без форка платформенной инфраструктуры.

### 7.1 Routing и URL-контракт

Обязательные маршруты:
- `astro.com/<new-product>` — человекочитаемый пользовательский маршрут (без версии в URL);
- `admin.astro.com/<new-product>` — человекочитаемый админ-маршрут (без версии в URL);
- `api.astro.com/v1/<new-product>/...` — версионируемый API-контракт (только сервисный контур).

Минимальный API-каркас:
- `POST /v1/<new-product>/calculate` (или эквивалент compute endpoint);
- `GET /v1/<new-product>/history`;
- `GET /v1/<new-product>/reports/:id` (если есть артефакт отчета).

Правила:
- пользовательские URL (`astro.com`, `admin.astro.com`) должны оставаться semantic/human-readable;
- не дублировать платформенные endpoint'ы (`/v1/auth`, `/v1/users/me`, `/v1/billing`);
- держать нейминг и версионирование консистентными с действующими модулями (`natal`, `numer`);
- `v1` относится только к стабильности API-контракта и не ограничивает изменение внутреннего pipeline модуля;
- фиксировать canonical URL и SEO-правила до запуска.

### 7.2 Auth и доступы (entitlements)

Подключение к единой сессии:
- cookie-модель доменного уровня (`Domain=.astro.com`, `HttpOnly`, `Secure`, `SameSite=Lax`);
- проверка пользователя только через единый session/user endpoint (`/v1/users/me`);
- единый logout/login flow без локальных auth-реализаций в модуле.

Подключение к entitlement-слою:
- завести product code модуля в общей карте тарифов;
- определить права вида `access_<new-product>`, `limit_<new-product>_*`;
- enforce прав и лимитов в API модуля (backend guard), а не только в UI.

Definition of Done для auth:
- пользователь с валидной общей сессией бесшовно входит в модуль;
- изменения тарифа/прав моментально отражаются в доступности сценариев;
- нет отдельной JWT/сессионной схемы внутри модуля.

### 7.3 Billing-интеграция

Обязательная интеграция:
- использовать только единый billing-контур и общий каталог тарифов;
- привязать SKU/plan модуля к существующим тарифам/аддонам;
- реализовать стандартные состояния: trial/free, active, grace, blocked/expired.
- не реализовывать отдельные payment flow/webhook хендлеры внутри модуля, если это платформенная функция.

Поведение paywall:
- если entitlement отсутствует, модуль переводит пользователя в единый upgrade flow;
- после успешной оплаты доступ пересчитывается централизованно и применяется без ручной синхронизации;
- все операции покупки/апгрейда/даунгрейда логируются в общем audit/event контуре.

Definition of Done для billing:
- покупка в рамках общего checkout открывает модуль без повторной покупки;
- история платежей и подписок отображается в общем кабинете;
- модуль не содержит собственных платежных интеграций.

### 7.4 Analytics и аудит

Минимальный event contract (обязательные события):
- `product_entry` (вход в модуль);
- `module_action_started`;
- `module_action_completed`;
- `module_report_generated` (если применимо);
- `paywall_shown`;
- `upgrade_started`;
- `upgrade_completed`;
- `module_error`.

Требования к событиям:
- единый формат: `user_id`, `session_id`, `product`, `event`, `timestamp`, `trace_id`, `source/utm`;
- сквозной `trace_id` между фронтом, API и фоновой обработкой;
- админ-действия по модулю пишутся в общий admin action log.

Definition of Done для analytics:
- воронка "entry -> first value -> paywall -> purchase/upgrade" собирается end-to-end;
- события сопоставимы между всеми модулями;
- есть дашборд по базовым KPI запуска.

### 7.5 Observability и NFR-политики

Обязательный минимум перед запуском:
- `health` и `readiness` checks для API-контура модуля;
- security headers + HSTS в соответствии с общей политикой;
- rate-limit и audit-логирование на критичных endpoint'ах;
- структурированные логи с `trace_id`, `user_id`, `product`;
- алерты на рост 5xx, деградацию latency, падение фоновых задач.

Если модуль генерирует артефакты (PDF/файлы):
- использовать общий `static`/storage-контур;
- статус пайплайна хранить в стандартизированных полях (`pipeline_stage`, `pipeline_error`);
- поддержать admin-retry сценарии для восстановлений без ручного вмешательства.
- допускается полностью кастомная последовательность шагов pipeline для каждого продукта при сохранении общего recovery-контракта.

Definition of Done для observability:
- инцидент можно диагностировать по логам/трейсам без доступа к коду;
- SLA/SLO метрики модуля видны в общей панели мониторинга;
- есть runbook для типовых сбоев (retry/report/email/storage).

- [ ] Реализована интеграция с общей сессией и `/v1/users/me`.
- [ ] Настроены entitlements и лимиты, добавлены backend guards.
- [ ] Подключен единый billing flow и paywall.
