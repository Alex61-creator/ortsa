# План перехода на `kerykeion` v5

Версия: `v1.1`  
Дата: `2026-04-16`  
**Статус документа:** `Partially closed (A/B pending)` — миграция на v5 и add-on MVP закрыты; **открыто:** A/B тесты цен и офферов (нет автоматического сплита вариантов цены/оффера в продукте).

Документ фиксирует:
- исторический as-is (`kerykeion` v4) и целевой слой v5;
- какие возможности открывает v5 и что уже внедрено;
- как безопасно мигрировать без регрессий в генерации отчетов (этап миграции завершён, см. §8.2).

---

## 1. Исходное состояние и контекст версий (исторически v4; зависимость сейчас v5)

### 1.1 Версия библиотеки в проекте

- `requirements.txt`: `kerykeion==5.0.4`
- `pyproject.toml`: `kerykeion = "^5.0.4"`

Итог: зависимость зафиксирована на v5; интеграция в коде переведена на factory API (см. §8.2). Ниже §1.2–§1.3 по-прежнему описывают прежнюю v4-схему и потенциал библиотеки, который ещё не выведен в отдельные продуктовые SKU.

### 1.2 Что именно используется в коде

Основная интеграция сосредоточена в `app/services/astrology.py`.

Используемые сущности:
- `AstrologicalSubject` (прямое создание субъекта);
- `KerykeionChartSVG` (генерация SVG для natal/synastry);
- `SynastryAspects` (аспекты синастрии);
- `Report` (текстовый отчет `get_full_report`);
- utility-функции для извлечения точек и домов.

Реальные сценарии использования:
- Натальная карта:
  - расчет положения планет/домов/углов;
  - генерация SVG/PNG колеса;
  - данные передаются в PDF и LLM pipeline.
- Синастрия:
  - двойное колесо;
  - аспекты между двумя людьми;
  - данные передаются в PDF и LLM pipeline.

Связанные пайплайны:
- `app/tasks/report_generation.py` (натальные отчеты);
- `app/tasks/synastry_generation.py` (синастрия).

### 1.3 Что сейчас не используется из экосистемы `kerykeion`

В проекте нет использования:
- `ChartDataFactory` + `ChartDrawer` (новая архитектура v5);
- `AspectsFactory` (single/dual chart aspects);
- transit/composite/returns;
- `RelationshipScoreFactory`;
- `MoonPhaseDetailsFactory`;
- `TransitsTimeRangeFactory`, `EphemerisDataFactory`;
- `to_context` (AI context serializer XML);
- расширенных опций рендера (темы, стили modern/classic, языковые пакеты, grid-only/wheel-only и т.д.).

#### Простыми словами: зачем это нужно

Этот список показывает не "что у нас плохо", а что мы пока не используем из потенциала библиотеки.

- `ChartDataFactory` + `ChartDrawer` нужны, чтобы разделить расчет астрологии и отрисовку, и за счет этого упростить поддержку и тестирование.
- `AspectsFactory` дает единый современный способ считать аспекты для одной и двух карт.
- transit/composite/returns - это новые типы отчетов, которые можно монетизировать как отдельные продукты.
- `RelationshipScoreFactory` добавляет понятную пользователю числовую оценку совместимости.
- `MoonPhaseDetailsFactory` дает расширенный лунный контекст для прогнозных и подписочных сценариев.
- `TransitsTimeRangeFactory` и `EphemerisDataFactory` позволяют строить прогноз не "на сейчас", а по диапазону дат.
- `to_context` стабилизирует входные данные для LLM и снижает риск пропусков/галлюцинаций в текстах.
- расширенные опции рендера нужны для более сильной визуальной упаковки отчетов и тарифной дифференциации.

#### Простыми словами по каждому пункту (готовая формулировка)

- `ChartDataFactory` + `ChartDrawer`  
  Нужны, чтобы разделить "математику астрологии" и "рисование колеса". Код становится чище, проще тестировать и меньше ломается при изменениях.

- `AspectsFactory`  
  Единый современный способ считать аспекты (и для 1 карты, и для 2 карт). Удобнее поддерживать, чем старые классы.

- transit/composite/returns  
  Это новые типы отчетов: транзиты, композит пары, соляр/лунар. По сути — дополнительные продукты/апселлы.

- `RelationshipScoreFactory`  
  Дает числовую оценку совместимости. Это удобно для карточки результата и маркетинга ("ваш score: 78/100").

- `MoonPhaseDetailsFactory`  
  Дает расширенные лунные данные (фаза, освещенность, ближайшие фазы и т.д.) — полезно для прогнозных и подписочных форматов.

- `TransitsTimeRangeFactory`, `EphemerisDataFactory`  
  Позволяют строить динамику во времени (не "срез на сейчас", а "что происходит по датам"). Нужны для календарей/таймлайнов/долгих прогнозов.

- `to_context`  
  Готовит структурированные данные для LLM в стабильном формате. Это уменьшает "разброс" качества текста и ошибки в интерпретациях.

- расширенные опции рендера (темы, modern/classic, языки, grid/wheel-only)  
  Нужны для более красивой упаковки PDF/изображений, локализации и разных визуальных вариантов под разные тарифы.

Итог: этот блок нужен, чтобы показать не "что у нас плохо", а какой потенциал мы не используем сейчас и почему переход на v5 — это не только техобновление, но и база для новых фич и выручки.

## 2. Что дает переход на v5 (to-be)

### 2.1 Архитектурные улучшения

Переход с монолитного legacy API на factory-подход:
- было (v4): `AstrologicalSubject` + `KerykeionChartSVG` + `SynastryAspects`;
- станет (v5):
  - `AstrologicalSubjectFactory` (создание субъекта),
  - `ChartDataFactory` (расчет структурированных данных),
  - `ChartDrawer` (визуализация SVG),
  - `AspectsFactory` (аспекты для single/dual chart).

Практический эффект:
- четкое разделение "расчет" vs "рендер";
- более предсказуемые типы данных;
- проще тестировать и кешировать результат расчетов.

### 2.2 Функции, которые можно добавить после миграции

#### Блок A: быстрое расширение текущего продукта

- **Transit chart**: сравнение натала с текущим небом (доп. продукт/подписка).
- **Composite chart**: отдельный relationship-отчет.
- **Solar/Lunar Return**: регулярные прогнозные отчеты.
- **Relationship score**: числовой KPI совместимости для тарифов/апселов.

#### Блок B: усиление AI/LLM контура

- **AI Context Serializer (`to_context`)**:
  - стабильная XML-структура фактов для промптов;
  - меньше "галлюцинаций" и логических пропусков в интерпретациях;
  - проще стандартизировать prompts между тарифами и языками.

#### Блок C: визуальная и продуктовая вариативность

- темы и стили (`classic` / `modern`);
- wheel-only / aspect-grid-only варианты для PDF и digital;
- языковые настройки рендера графики;
- более гибкая визуальная упаковка под разные тарифы.

#### Простыми словами: что это дает продукту

- Блок A про быстрый запуск новых форматов отчетов и апселов без переписывания базовой астрологической логики.
- Блок B про повышение качества LLM-интерпретаций: модель получает структурированные факты и работает стабильнее.
- Блок C про упаковку и UX: больше визуальных вариантов под разные тарифы, языки и каналы (PDF/digital).

Итог: переход на v5 - это не только техническая миграция, но и база для роста продуктовой линейки, качества контента и выручки.

### 2.3 Ожидаемые преимущества для бизнеса и операционки

- **Скорость вывода новых тарифов**: transit/return/composite без кастомной математики.
- **Выручка**: больше продуктовых SKU и апселов (monthly transit, compatibility score, returns).
- **Качество отчета**: более структурированные данные для LLM -> более стабильные тексты.
- **Сопровождаемость**: меньше техдолга вокруг legacy API и ниже риск "сломаться" при будущих апдейтах.
- **Тестируемость**: удобнее покрыть unit/integration тестами фабрики и рендер отдельно.

## 3. План миграции (по этапам)

## 3.1 Этап 0 — подготовка (без смены логики)

- Зафиксировать baseline:
  - golden-образцы текущих PNG/SVG/PDF (natal + synastry);
  - snapshot JSON-данных, уходящих в LLM;
  - метрики времени генерации и частоты ошибок.
- Добавить smoke-тесты пайплайнов:
  - `report_generation` (natal),
  - `synastry_generation` (synastry).

Критерий выхода: есть с чем сравнивать после апгрейда.

## 3.2 Этап 1 — upgrade зависимости и совместимость

- Обновить зависимость до v5.
- Временно оставить backward-compatible слой только там, где нужно быстро стабилизироваться.
- Пройтись по breaking changes из migration-guide:
  - `AstrologicalSubject` -> `AstrologicalSubjectFactory.from_birth_data`;
  - `KerykeionChartSVG` -> `ChartDataFactory` + `ChartDrawer`;
  - `SynastryAspects` -> `AspectsFactory.dual_chart_aspects`.

Критерий выхода: проект компилируется, базовые сценарии запускаются локально.

## 3.3 Этап 2 — рефактор `app/services/astrology.py`

Новая целевая структура сервиса:
- `build_subject(...)` через `AstrologicalSubjectFactory`;
- `build_natal_chart_data(...)` / `build_synastry_chart_data(...)` через `ChartDataFactory`;
- `render_svg(...)` через `ChartDrawer`;
- `calculate_aspects(...)` через `AspectsFactory`.

Задача этапа: сохранить текущий контракт сервиса, чтобы внешние задачи (`report_generation`, `synastry_generation`) не менять массово за один шаг.

Критерий выхода: API сервиса совместим, бизнес-логика не меняется.

## 3.4 Этап 3 — выравнивание данных для PDF и LLM

- Сопоставить поля v4 и v5 в контексте шаблонов `report*.html`, `synastry.html`.
- Проверить, что все нужные поля (`planets`, `houses`, `angles`, `aspects`) присутствуют.
- Для LLM-пайплайна добавить опциональную ветку `to_context` (feature flag), не ломая текущие промпты.

Критерий выхода: PDF и LLM тексты не деградируют по полноте и формату.

## 3.5 Этап 4 — стабилизация и cleanup

- Прогон regression pack:
  - сравнение golden PNG/SVG/PDF;
  - проверка ключевых секций в текстах;
  - нагрузочный прогон Celery задач.
- Удалить legacy-вызовы и временные адаптеры.
- Зафиксировать новый стандарт интеграции в техдоках.

Критерий выхода: v5 полностью в продовом контуре, legacy API не используется.

## 4. Риски и как ими управлять

- **Риск несовпадения структуры данных** (v4 vs v5):  
  Митигировать адаптером поля и snapshot-тестами контекста PDF/LLM.

- **Риск визуальных отличий колеса**:  
  Митигировать golden-сравнениями артефактов и явно зафиксировать тему/стиль.

- **Риск просадки времени генерации**:  
  Митигировать бенчмарком до/после и профилированием в Celery.

- **Риск регрессий в аспектах**:  
  Митигировать тестами на эталонных натальных парах.

## 6. Новые фичи для текущих тарифов

Ниже — прикладной вариант, как встроить новые возможности `kerykeion` в уже существующую тарифную линейку без полной переделки продукта.

### 6.1 Тариф `FREE` (входной)

- **Что добавить**:
  - mini transit-блок "энергия ближайших 7 дней";
  - упрощенный lunar-блок (фаза луны + короткий совет);
  - более выразительная визуализация (`modern` wheel-only).
- **Что это даст**:
  - выше конверсия из `FREE` в платные тарифы за счет "вкуса прогноза";
  - выше вовлеченность даже у бесплатных пользователей;
  - более сильный вау-эффект от первого отчета.

### 6.2 Базовый платный тариф (персональный natal)

- **Что добавить**:
  - расширенный раздел transit на 30 дней;
  - monthly ephemeris-таймлайн по ключевым точкам;
  - структурированный `to_context` для стабильного качества текста.
- **Что это даст**:
  - рост perceived value без резкого удорожания генерации;
  - меньше претензий к "разбросу" качества интерпретаций;
  - аргумент для повышения цены базового тарифа.

### 6.3 Тариф с синастрией (`SYNASTRY_BUNDLE` / аналог)

- **Что добавить**:
  - `RelationshipScoreFactory` (score + краткая расшифровка);
  - composite-отчет как дополнительный раздел;
  - аспекты через `AspectsFactory` с более единообразным форматом.
- **Что это даст**:
  - понятная метрика совместимости для пользователя ("score");
  - новый апселл: "подробный composite";
  - улучшение повторных покупок в парных сценариях.

### 6.4 Подписочный/премиальный тариф (`PRO` / годовой)

- **Что добавить**:
  - solar return (годовой цикл) + lunar return (ежемесячный ритм);
  - транзиты по диапазону дат (`TransitsTimeRangeFactory`);
  - прогнозный календарь с пиками и "окнами возможностей".
- **Что это даст**:
  - сильный подписочный value proposition;
  - регулярные причины возвращаться в продукт каждый месяц;
  - снижение оттока за счет контента с "продолжением".

### 6.5 Add-on механики (поверх текущих тарифов)

- **Что добавить**:
  - "Transit month pack";
  - "Compatibility deep dive" (score + composite + расширенная синастрия);
  - "Return pack" (solar + 3 lunar отчета).
- **Что это даст**:
  - дополнительная выручка без изменения ядра тарифа;
  - гибкая монетизация для разных сегментов аудитории;
  - удобные маркетинговые офферы для email/push-кампаний.

#### Подробно: что это и зачем

- **Transit month pack**
  - Что это: дополнительный 30-дневный прогноз поверх уже купленного отчета (ключевые транзиты, даты напряжения/возможностей, короткие рекомендации).
  - Для чего: дать "продолжение" после базового отчета и конвертировать разовый платеж в повторный.
  - Это мост к подписке: да, по сути это первый шаг перевода покупателей разового отчета в регулярное потребление. Пользователь сначала пробует один "месяц прогноза", после чего проще продавать auto-renew подписку "ежемесячный транзит".
  - Кому продавать: пользователям с завершенным natal-отчетом через 3-7 дней после покупки.

- **Compatibility deep dive**
  - Что это: расширенный парный пакет: `Relationship score` + `Composite chart` + расширенная синастрия (больше аспектов и практический блок "как улучшать взаимодействие").
  - Для чего: повысить средний чек в парном сценарии и дать более осязаемый результат, чем "только текст совместимости".
  - Кому продавать: пользователям, уже купившим синастрию или находящимся в сценарии "пара/отношения".

- **Return pack**
  - Что это: прогнозный пакет на цикл: `Solar return` + 3 последовательных `Lunar return` отчета.
  - Для чего: формировать подписочную привычку и удержание (регулярные точки контакта каждый месяц).
  - Кому продавать: активным клиентам с повторными визитами, особенно после высоких NPS/позитивных оценок.

##### Механика взаимодействия с пользователем для `Return pack`

- **Когда касаться**:
  - основной оффер: через 24-72 часа после прочтения базового natal-отчета;
  - повторный оффер: за 7 дней до дня рождения (контекст solar return);
  - reactivation-оффер: спустя 21-30 дней неактивности.
- **Как касаться**:
  - в кабинете: баннер/карточка "Ваш годовой цикл + 3 лунных отчета";
  - email: цепочка из 2-3 писем (польза -> пример инсайта -> CTA на покупку);
  - push/telegram: короткое напоминание в день старта лунного периода.
- **Для чего в каждый момент**:
  - post-report оффер: конверсия "горячего" интереса в первую допокупку;
  - pre-birthday оффер: максимальная релевантность и готовность платить;
  - reactivation: вернуть пользователя в продукт новым контентным поводом.
- **К чему приведет**:
  - рост retention за счет регулярного контента;
  - рост LTV за счет пакета и последующей подписки;
  - снижение зависимости от одноразовых покупок.

#### Как внедрить в проект для допродаж

1. **Добавить тарифные сущности (без ломки текущих)**
   - Ввести новые `tariff.code`: `transit_month_pack`, `compatibility_deep_dive`, `return_pack`.
   - Хранить их как add-on тарифы (аналогично текущей механике `synastry_addon`), чтобы не переписывать основной checkout.
   - Для каждого add-on задать: цену, доступные артефакты, лимиты, срок действия.

##### Как связать add-on с текущими тарифами и не сломать основную логику

- **Принцип интеграции**:
  - базовые тарифы (`FREE`, natal, synastry bundle и т.д.) остаются без изменений;
  - add-on оформляется как отдельный заказ, но с ссылкой на "родительский" базовый отчет/заказ;
  - текущий pipeline оплаты, webhook и статусы заказа переиспользуются.
- **Модель связи**:
  - в `orders` добавить поле-референс `parent_order_id` (или отдельную связующую таблицу `order_addons`);
  - add-on может быть куплен только если `parent_order` в статусе `COMPLETED`;
  - запрет дублей: один и тот же add-on не продается повторно для того же `parent_order`, если это не предусмотрено бизнес-правилами.
- **Совместимость с текущими тарифами**:
  - `transit_month_pack` доступен для пользователей с завершенным natal-отчетом;
  - `compatibility_deep_dive` доступен для завершенной синастрии;
  - `return_pack` доступен для завершенного natal-отчета и активного аккаунта.
- **Где и на каком этапе предлагать**:
  - сразу после генерации и просмотра отчета: in-app CTA в карточке отчета;
  - в `dashboard/reports`: блок "Улучшить отчет" с персональным оффером;
  - email/push после события "report_completed" (отложенный триггер 24-72 часа);
  - при повторном входе в кабинет: sticky-оффер до покупки/dismiss.
- **Почему это не ломает логику**:

## 7. Что меняется в `PRODUCT_FLOW_AND_BUSINESS_LOGIC.md`

Ниже зафиксировано, что добавится в текущую продуктовую схему (документ `docs/PRODUCT_FLOW_AND_BUSINESS_LOGIC.md`) после перехода на `kerykeion` v5 и запуска add-on тарифов.

### 7.1 Новое в ключевых сущностях

К существующим сущностям из `PRODUCT_FLOW_AND_BUSINESS_LOGIC.md` добавляются:
- `AddOnTariff` (логически в рамках `Tariff`): тарифы допродажи (`transit_month_pack`, `compatibility_deep_dive`, `return_pack`);
- `AddOnOrder` (логически в рамках `Order`): отдельный заказ допродажи, связанный с базовым заказом;
- `AddOnReport` (логически в рамках `Report`): PDF/артефакты add-on отчета, привязанные к `parent_order_id`.

Примечание по реализации: физически можно не вводить новые таблицы на первом этапе, а расширить текущие `tariffs/orders/reports` (через `tariff.code` и связь с родительским заказом).

### 7.2 Новое в End-to-End клиентском пути

Текущий путь "лендинг -> оплата -> генерация -> выдача отчета" сохраняется, добавляется post-purchase контур:

10. После просмотра базового отчета пользователь видит персональный add-on оффер в кабинете.  
11. При покупке add-on создается связанный заказ допродажи (без изменения базового заказа).  
12. webhook переводит add-on заказ в `PAID` и запускает отдельную Celery-задачу генерации add-on отчета.  
13. Пользователь получает add-on отчет в том же канале (dashboard + email), где получает базовые отчеты.

### 7.3 Новое в API-областях

К каноническим API-областям добавляется:
- `/api/v1/addons/*` — покупка, статус и выдача add-on отчетов.

Минимальный набор ручек (MVP):
- `POST /api/v1/addons/transit-month/purchase`
- `POST /api/v1/addons/compatibility-deep-dive/purchase`
- `POST /api/v1/addons/return-pack/purchase`
- `GET /api/v1/addons` (доступные офферы для пользователя)
- `GET /api/v1/addons/{order_id}` (статус/метаданные)

### 7.4 Новое в правилах и ограничениях

Добавляются бизнес-правила:
- add-on доступен только при `COMPLETED` базовом заказе;
- add-on доступен только владельцу базового заказа;
- для части add-on действует ограничение по времени оффера (например, 72 часа);
- запрет дублей для одного и того же add-on к одному `parent_order_id` (если не разрешено отдельно).

## 9. Executive summary

Проект на **v5 factory API** в расчётах/рендере натала и синастрии; **часть** возможностей `kerykeion` (transit/composite/returns/scoring и т.д.) ещё не выведена в отдельные продуктовые отчёты — см. §2 и §5–§6.  
Завершённый переход на v5 даёт:
- более устойчивую архитектуру расчетов/рендера;
- снижение техдолга и рисков дальнейших обновлений;
- фундамент для новых продуктовых направлений (transits/returns/composite/scoring);
- более стабильный и масштабируемый LLM-контур через структурированные данные.

Миграцию на v5 выполняли поэтапно с golden-тестами (см. архив regression pack в §8.2). Дальнейший рост — через новые SKU (§5–§6) и закрытие open-loop из §8.3 (A/B).
